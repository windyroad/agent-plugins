# Project: windyroad-claude-plugin

Plugin-development monorepo publishing `@windyroad/*` Claude Code plugins — architecture governance, risk management, ITIL, TDD, JTBD, retrospectives, and delivery quality — by Windy Road Technology. These plugins promote Windy Road's service offering to the community; they are NOT internal project utilities.

Not a web UI project — accessibility-first global guidance (injected by the `accessibility-agents` plugin into `~/CLAUDE.md`) does not apply here.

**MANDATORY — act on obvious, AskUserQuestion for ambiguous, NEVER prose-ask** (P085): when the user has pinned a direction (yes / go / proceed / act / just do it) or the next step is obvious from session context or RISK-POLICY.md appetite, act and report — do not surface a consent gate. When genuinely ambiguous, use the `AskUserQuestion` tool. Prose asks ("Want me to...?", "Should I...?", "Option A or Option B?", "(a)/(b)/(c)?") are non-compliant and unanswerable under AFK notifications. Canonical phrasing list + hook enforcement: `packages/itil/hooks/lib/detectors.sh`.

**MANDATORY — capture on correction** (P078): when the user delivers a strong-signal correction (FFS / all-caps DO NOT / direct contradiction / `!!!` / "you always|never|keep"), OFFER to capture a problem ticket via `/wr-itil:capture-problem` (or `/wr-itil:manage-problem` fallback) BEFORE addressing the operational request. The correction is a class-of-behaviour signal, not a one-off; a ticket preserves it as durable WSJF-ranked backlog. Pattern vocabulary + hook enforcement: `packages/itil/hooks/lib/detectors.sh::CORRECTION_SIGNAL_PATTERNS` + `packages/itil/hooks/itil-correction-detect.sh`.

**MANDATORY — never write project-generated artefacts under `.claude/`** (P131): `.claude/` is user-controlled config space (settings, MEMORY.md, MCP server configs, user-authored hooks, user-authored skills). Project-generated content — implementation plans, audit logs, scratch state, briefing artefacts, agent-output dumps — belongs under `docs/` (e.g. `docs/plans/`, `docs/audits/`, `docs/briefing/`) or directly inline in problem-ticket bodies. Gate-exclusion lists (architect, JTBD, TDD, style-guide, voice-tone, risk-scorer hooks) that name `.claude/` paths mean **the gate doesn't fire on user edits to those paths** — they do NOT mean agents may write there. The exclusion is READ tolerance, not WRITE permission. If a long-running plan or audit must be persisted across session boundaries, write it to `docs/plans/` (or attach inline to the relevant problem ticket); never to `.claude/plans/` or any other agent-invented `.claude/` subpath.

**MANDATORY — when a SKILL contract names a stage as mechanical, do not ask** (P132 / inverse-P078): when a SKILL contract explicitly carves out a stage as **mechanical** / **no user decision** / **policy-authorised silent proceed** / **agent owns silent classification** / **silent agent action**, do NOT call `AskUserQuestion` in that stage. The framework has already resolved the decision; per-action consent gates re-ask decisions the user already made and silently undo the load-bearing UX investment of the mechanical-stage carve-out. Save `AskUserQuestion` for the 6-class authority taxonomy in ADR-044 — direction-setting / deviation-approval / one-time-override / silent-framework / taste / authentic-correction. Defensive over-asking accumulated from upstream user corrections (the inverse-P078 trap) is the typical failure mode: agent gets corrected on N decisions, agent over-generalises Rule 1 to "ask before deciding ANYTHING", agent silently re-introduces friction the SKILL was engineered to remove. Worked examples already in-skill: `run-retro` Step 1.5 silent classification, Step 3 briefing removals, Step 3 Tier 3 topic-file rotation, Step 4a verification close-on-evidence, Step 4b Stage 1 mechanical ticketing + Stage 2 silent fix-strategy pick; `/wr-itil:work-problems` Step 6.5 within-appetite drain + ADR-042 auto-apply; `/install-updates` Step 6a cache hit. Authority: `docs/decisions/044-decision-delegation-contract.proposed.md` framework-resolution boundary; lazy-AskUserQuestion-count is the regression metric per Step 2d Ask Hygiene Pass (`packages/retrospective/scripts/check-ask-hygiene.sh`).

**MANDATORY — user direction is not substance ratification; brief-and-ratify AFTER governance edits** (P357): when you apply a user-directed edit to a governance artefact (ADR / JTBD / persona / RFC / story) given as freeform prose *outside* the `/wr-architect:create-adr` Step 5, `/wr-architect:review-decisions`, or `/wr-jtbd:confirm-jobs-and-personas` flows (which already carry the gate), user direction does NOT by itself authorise the `human-oversight: confirmed` marker. The implication "user directed it → substance ratified" holds ONLY if the LLM understood the direction correctly and implemented it without semantic drift — not guaranteed (wrong mechanism, scope expansion, missed constraint). AFTER applying the edit and BEFORE writing the marker (or invoking `wr-architect-mark-oversight-confirmed` / `wr-jtbd-mark-oversight-confirmed`), surface a self-contained post-change brief of what actually changed in plain prose — never cryptic IDs as the carrier of meaning (P350) — and confirm the LLM's interpretation via `AskUserQuestion`. Under AFK (no `AskUserQuestion`), write `human-oversight: unconfirmed` and queue the post-change ratification for the next interactive drain (ADR-066 P348 fallback). Calling the marker shim citing "user direction = ratification" without a real post-change confirm event is the P348 hollow-marker bug on the freeform-amendment path. Sibling to P340 (the AskUserQuestion-path gap, closed at `/wr-architect:create-adr` Step 5 § 5a). The structural enforcement mechanism (hook / interactive-shim / behavioural-test) is a deferred design decision tracked in P357.
