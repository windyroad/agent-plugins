### What worked

- **Plan was already in P081 ticket body** — the Implementation Plan section drafted 2026-04-27 by the Plan agent gave explicit §1-§8 step decomposition + critical files list + architectural trade-offs + implementation order. Iter 13 followed the plan with one numbering correction (ADR-044 → ADR-052 since 044 was taken). Pre-baked plans inside Open tickets are high-leverage when the ticket reaches the top of the queue weeks later — the agent picking up the ticket starts from "review the plan + architect re-confirm + execute" rather than "decompose the work + architect-from-scratch + execute". This is the codification candidate that emerged from iter 13: when WSJF picks an Effort-L ticket whose body already contains an Implementation Plan section, the iter is ~30% lower-friction than a same-effort ticket without one.
- **Architect two-round verdict** — the initial 6-issues-found round was high-density actionable feedback (ADR numbering correction, supersession-banner mirroring, Migration section, ADR-044 escape-hatch category citations, ADR-026 grounding for harness_gap, ADR-005 narrowing as additive sub-clause). All 6 issues were incorporated in a single revision round; the re-confirm returned PASS with no new issues. Pattern: when architect issues are concrete file-paths + corrective-actions, single-pass incorporation is achievable; when they're abstract framing concerns, multi-pass is the norm.
- **Behavioural-only bats fixture as dogfood** — the bats fixture for `tdd-review-test.sh` exercises the hook with mock JSON tool input and asserts on emitted output (or its silence) across all six advisory paths, with NO greps of either `tdd-review-test.sh` source or `review-test.md` agent source. This makes the fixture itself ADR-052-compliant on day one — the canonical behavioural-default exemplar plugin authors copy.

### What didn't work

- **ADR numbering collision** — the Plan section in P081's body named the new ADR as ADR-044, but ADR-044 was taken by `decision-delegation-contract.proposed.md` (filed after the plan was drafted 2026-04-27). The collision was caught at the architect-review step (the prompt itself flagged "ADR-052 (NOT 044 — 044 is now taken … 052 is next available)"), so the cost was a single grep + bash `ls` to find the highest current ADR number. Pattern: pre-baked Implementation Plans that name specific ADR/JTBD/P-IDs decay over time as the index advances — the plan author should consider naming the artefact by description rather than ID, OR the executing agent should always re-verify ID availability against `ls docs/decisions/` before using the planned ID.
- **The `architect-enforce-edit.sh` PreToolUse Write gate fired on the first ADR-052 write** despite an earlier same-session architect-review marker existing. The marker had been written by the initial architect run, but the gate's per-file logic required a re-confirm before allowing the file to land. The re-confirm round (~90s) returned PASS cleanly, but the apparent two-round architect cost was 6 ISSUES → fix → re-confirm PASS → write. Pattern: when an architect review reports ISSUES FOUND, the hook gate may treat the marker as "advisory not yet honoured" and require a re-confirm even after issue-incorporation; the second round is rarely friction (concrete-issue fixes are cheap to confirm) but is an extra ~90s.
