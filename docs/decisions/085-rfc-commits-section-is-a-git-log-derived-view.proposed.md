# The RFC `## Commits` section is a git-log-derived view, rendered skill-side

## Context and Problem Statement

ADR-060 Phase 1 item 12 promised that an RFC's `## Commits` section would be auto-maintained from the commit-message `Refs: RFC-NNN` trailer. The executor was deferred ("lands in Slice 3 task B5.T9") and never built (P378). capture-rfc's template therefore wrote a false "(maintained automatically)" claim over an empty section, while trailer-bearing commits went unrecorded — the RFC framework leaning on human memory, the anti-pattern the repo exists to prevent (P375/P377).

The obvious fix — a PostToolUse hook that edits the RFC after each commit — is barred: a post-commit working-tree edit lands OUTSIDE the commit it describes, fragmenting the ADR-014 single-commit grain (the architect's prior Q2 verdict already settled this for the sibling `## RFCs` surface).

## Decision Drivers

- **No post-commit working-tree edits** (ADR-014 grain).
- **Self-maintaining, not memory-bound** (P378 / P375 / P377).
- **Reuse the established rendered-index pattern** (ADR-031: `docs/problems/README.md` is a view derived from the per-ticket files).
- **Honesty** — never claim automation that doesn't exist (P234 fictional-defer).

## Considered Options

1. **PostToolUse hook edits the RFC after commit** — rejected: violates ADR-014 grain (the edit is uncommitted churn).
2. **Derived view, rendered skill-side** — `## Commits` is a projection of `git log --grep "Refs: RFC-NNN"`, regenerated by `manage-rfc` (every transition/review) + `reconcile-rfcs`. Nothing stored per-commit.
3. **Derived view + a SessionStart drift-detector** — option 2 plus a read-only SessionStart hook flagging stale sections (cloning the ADR-084 self-firing shape).

## Decision Outcome

Chosen option: **Option 2 — derived view, rendered skill-side** (user-pinned via AskUserQuestion, 2026-06-25). The commit log is the source of truth; the `## Commits` section is a projection rendered by `packages/itil/scripts/update-rfc-commits-section.sh` (`git log --grep "Refs: RFC-NNN"`), invoked by `manage-rfc` on every transition/review and by `reconcile-rfcs`. Because nothing is written per-commit there is no ADR-014 grain problem. Between renders the section is *honestly* stale (rendered-as-of-last-skill-run), never a false "maintained automatically".

Option 3's SessionStart drift-detector is deferred on the architect's evidence-based lean: add it only if `## Commits` staleness is observed to rot in practice (mirrors how ADR-084's self-firing census was justified by measured rot, not assumed). That deferral is itself surfaced — RFC-030 is tracked and self-surfaced by the RFC-oversight nudge — so it is not an uncadenced deferral (P375).

## Consequences

### Good
- Closes the never-built ADR-060 item 12 without an ADR-014 violation; RFC `## Commits` is accurate after any skill run (verified: RFC-029 now lists all 7 of its commits).
- Directly mirrors ADR-031, so the mental model is consistent across the problem/RFC surfaces.

### Bad
- Between skill runs the section can lag the true git state (honest staleness). The drift-detector (Option 3) is the mitigation if this proves to matter.

## Pros and Cons of the Options

### Option 1 — post-commit hook
- Bad: ADR-014 grain violation (uncommitted post-commit edit). Settled against by the prior Q2 verdict.

### Option 2 — derived view, skill-side (chosen)
- Good: no per-commit write; ADR-031 precedent; honest staleness.
- Bad: lags between skill runs (mitigable via Option 3 if evidenced).

### Option 3 — derived view + SessionStart drift-detector
- Good: closes the never-re-rendered window (ADR-084 shape).
- Bad: another hook; add only on evidence of rot (architect lean).

## Reassessment Criteria

Revisit if `## Commits` staleness is observed to mislead in practice (→ add Option 3's drift-detector), or if a per-commit source-of-truth becomes necessary.
