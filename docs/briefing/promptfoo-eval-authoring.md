## What You Need to Know

### AGENT-prose evals (RFC-012 S1) run the agent via `claude -p --system-prompt "$(cat agent.md)"` (full replace, NOT `--append`) and cd to repo root; headless stdout is non-deterministic — Tier-A hard tokens (job IDs, verdict-template headers) flake ~50%, anchor Tier-A on the verdict-OUTCOME token only (2026-06-27)

The agent-prose twin of the SKILL evals (P324). Driver shape (`packages/jtbd/agents/eval/run-agent-eval.sh`): resolve repo root, `cd` there (so the agent's live `docs/jtbd/` reads + the `wr-jtbd-is-job-or-persona-unconfirmed` PATH shim's default root resolve), then `exec claude -p --system-prompt "$(cat ../agent.md)" "$@"`. Use **`--system-prompt` (full replace)** for an agent eval — its prose IS the entire instruction set — distinct from the SKILL evals' `--append-system-prompt` (ADR-075). **Key flake lesson:** under headless `claude -p`, the agent frequently writes its full verdict to the marker file (`/tmp/jtbd-verdict`) and emits only a TERSE summary to stdout — so Tier-A `icontains` on a specific job ID (`JTBD-006`) or the verdict-template header (`JTBD Review`) flaked ~50% across runs even though the verdict was behaviourally correct. **Fix:** anchor Tier-A on the verdict-OUTCOME token alone (`icontains: PASS`) — it survives terse + full forms — and route job-naming + all negative-clause semantics (the over-fire `[Unratified Dependency]` guard) to Tier-B `llm-rubric` (which tolerates phrasing variance). Verify GREEN over ≥3 consecutive runs before committing; single-sample pass^k robustness is RFC-012 S3's job, not S1's. Worked example: `packages/jtbd/agents/eval/promptfooconfig.yaml`. Adjacent: both governance oversight drains have CLOSED (0 unratified jtbd jobs/personas + 0 unratified ADRs as of 2026-06-27), so a live positive-fire `[Unratified Dependency]` fixture is no longer constructable against real docs — it needs a synthetic fixture corpus (the shim accepts an optional `[JTBD_DIR]` second arg). <!-- signal-score: 0 | last-classified: 2026-06-27 | first-written: 2026-06-27 -->

### Tarball-excluding a per-package eval dir: a `.npmignore` does NOT exclude a subdir of a `files`-allowlisted dir — use a `files`-field NEGATION entry (`!agents/eval/`) (2026-06-27)

When `package.json` has a `files` allowlist that includes a parent dir (e.g. `agents/`), a sibling `.npmignore` listing `agents/eval/` does NOT remove it from the tarball — npm includes the allowlisted dir wholesale. The working mechanism is a **negation entry in the `files` array itself**: add `"!agents/eval/"` after the allowlist entries (exactly as `packages/itil/package.json` does with `"!skills/*/eval/"`). The itil `.npmignore` that claims to be the exclusion mechanism is redundant/misleading — the `files`-negation is what actually fires. Verify with `npm pack --dry-run` from the package dir: no `agents/eval/` path should appear (P324 S1, ADR-075 Confirmation §1 tarball-exclusion). **Re-confirmed independently for `skills/*/eval/` on the FIRST risk-scorer eval (P381, 2026-06-28): adding `packages/risk-scorer/.npmignore` with `skills/*/eval/` changed nothing; moving `packages/itil/.npmignore` aside did NOT make itil's eval ship — the decisive test. The fix was `"!skills/*/eval/"` in `packages/risk-scorer/package.json` `files`. CAUTION: the `wr-architect:agent` gate itself initially mis-attributed the mechanism to the `.npmignore` — verify empirically with `npm pack --dry-run`, never trust the `.npmignore` presence (or the architect's reasoning about it) as proof. When adding the FIRST eval under a plugin with no `.npmignore`, you MUST add the `files`-negation or the eval ships.** <!-- signal-score: 2 | last-classified: 2026-06-28 | first-written: 2026-06-27 -->

### A flaky SKILL eval suite usually has MULTIPLE brittle cases, not one — and not every flake is Tier-A: the Tier-B rubric can CORRECTLY fail a genuine SKILL-prose self-contradiction (2026-06-27)

Two lessons from P393 (run-retro eval, 6/7-intermittent). (1) **Don't whack-a-mole one case — audit the whole suite.** When a suite reports "1 failed" but a re-run is green, it is FLAKY, not deterministically red. Each case carrying multi-part / negative-clause Tier-A regexes flakes ~1-in-3 independently, so for a 5-such-case suite P(all green) ≈ (2/3)^5 ≈ 0.13 — you will keep hitting a *different* failing case each run. Fix once: demote the semantic / negative-clause Tier-A regexes across EVERY case to the (already comprehensive) Tier-B `llm-rubric`, keeping Tier-A only for distinctive paraphrase-proof positive anchors (numeric thresholds, command names, citation / classification tokens). (2) **When demoting Tier-A doesn't stabilise a case, read WHICH assert fails before assuming brittleness.** In P393 the residual flaky case (Step 1.5) had both retained Tier-A anchors PASSING and the **Tier-B llm-rubric FAILING** — because the model gave a genuinely *wrong* answer ~1-in-3. Root cause was a real SKILL.md self-contradiction (Step 1.5 said `<=-3` deletes fire a batched `AskUserQuestion`; Step 3 + CLAUDE.md P132 + ADR-044 said briefing removals are silent). The model nondeterministically followed the stale prose. The eval was working as designed — it caught a SKILL defect. **Fix the SKILL prose, never weaken the rubric to accept the wrong answer.** Use `--output <json>` + parse `gradingResult.componentResults` to see per-assert pass/fail; a failing Tier-B with passing Tier-A is the tell that the SKILL-under-test is wrong, not the eval. Verify GREEN over ≥3 consecutive runs (LLM non-determinism) before calling it stable. <!-- signal-score: 0 | last-classified: 2026-06-27 | first-written: 2026-06-27 -->

### For a SKILL-prose behavioural test, author a promptfoo eval case — NOT a structural bats fixture: ADR-052's 2026-06-09 amendment DELETED the structural escape hatch (no `tdd-review: structural-permitted` justification works any more), even when mirroring the existing P146/P232 grep fixtures (2026-06-28)

When a SKILL.md prose change needs a behavioural second-source, reach for a **promptfoo eval case** (add it to the skill's `eval/promptfooconfig.yaml`), NOT a `grep`-on-SKILL.md bats fixture — even though many such structural fixtures already exist in-tree (`work-problems-step-5-bats-polling-discipline.bats` etc.). Those pre-existing fixtures are **tracked debt** (P290 Phase 2 / blocked on P324), recorded as a known state-of-violation — NOT a pattern to copy. ADR-052's **2026-06-09 amendment (Option 1A, behavioural-only, no escape hatch)** removed the former "Surface 2" / structural-permitted carve-out in its entirety: structural assertions on prose-document content (`SKILL.md`, `agent.md`, ADR bodies) are "not permitted under any justification". ADR-037 (which those fixtures cite as their "Permitted Exception" authority) is **superseded by ADR-052**. The `wr-architect:agent` edit-gate returns ISSUES FOUND on a net-new structural SKILL-prose grep fixture and withholds the marker (it only sets the marker on a PASS verdict — `architect-mark-reviewed.sh`), blocking the edit. The clean path: a promptfoo case that exercises the skill-shape and asserts on the response (Tier-A paraphrase-proof anchors + Tier-B llm-rubric) — which ALSO discharges the R009 in-source floor for the prose change per ADR-061 Rule 4.
