## What You Need to Know

### For a SKILL-prose behavioural test, author a promptfoo eval case — NOT a structural bats fixture: ADR-052's 2026-06-09 amendment DELETED the structural escape hatch (no `tdd-review: structural-permitted` justification works any more), even when mirroring the existing P146/P232 grep fixtures (2026-06-28)

When a SKILL.md prose change needs a behavioural second-source, reach for a **promptfoo eval case** (add it to the skill's `eval/promptfooconfig.yaml`), NOT a `grep`-on-SKILL.md bats fixture — even though many such structural fixtures already exist in-tree (`work-problems-step-5-bats-polling-discipline.bats` etc.). Those pre-existing fixtures are **tracked debt** (P290 Phase 2 / blocked on P324), recorded as a known state-of-violation — NOT a pattern to copy. ADR-052's **2026-06-09 amendment (Option 1A, behavioural-only, no escape hatch)** removed the former "Surface 2" / structural-permitted carve-out in its entirety: structural assertions on prose-document content (`SKILL.md`, `agent.md`, ADR bodies) are "not permitted under any justification". ADR-037 (which those fixtures cite as their "Permitted Exception" authority) is **superseded by ADR-052**. The `wr-architect:agent` edit-gate returns ISSUES FOUND on a net-new structural SKILL-prose grep fixture and withholds the marker (it only sets the marker on a PASS verdict — `architect-mark-reviewed.sh`), blocking the edit. The clean path: a promptfoo case that exercises the skill-shape and asserts on the response (Tier-A paraphrase-proof anchors + Tier-B llm-rubric) — which ALSO discharges the R009 in-source floor for the prose change per ADR-061 Rule 4. Witnessed P370 (2026-06-28): the iter task brief specified the structural-bats option as a primary choice (brief written against pre-amendment ADR-052); the architect gate caught it and the fix pivoted to a promptfoo case (GREEN, 1 passed). <!-- signal-score: 0 | last-classified: 2026-06-28 | first-written: 2026-06-28 -->

### Don't anchor Tier-A `contains <skill-name>` when the eval prompt is already INSIDE that skill's invocation — the response describes the path without re-naming the skill (2026-06-29)

P270-class brittleness, fresh instance (P399 capture-rfc eval). A `contains: capture-rfc` Tier-A anchor false-failed a behaviourally-CORRECT answer: the eval prompt framed the model as already invoking `/wr-itil:capture-rfc --fix-time`, so the response answered *about the `--fix-time` path* ("on the `--fix-time` path you write fully-authored content…") without echoing the skill name. The Tier-B rubric PASSED (substance correct); only the brittle skill-name `contains` failed. **Rule: the distinctive paraphrase-proof anchor for a flag-driven branch is the FLAG (`--fix-time`) + the structural headings it authors (`## Scope` / `## Tasks`), not the skill name** — the skill name is only a reliable anchor when the prompt asks "name the exact command I run" (e.g. the work-problems Step-5 gate-invocation case, where the response *does* emit `/wr-itil:capture-rfc --fix-time`). Same calibration lesson as the work-problems P390 case dropping its over-tight negation regex. Diagnose with `--output <json>` + `gradingResult.componentResults` (passing Tier-A flag anchor + passing Tier-B rubric + the lone failing `contains skill-name` is the tell). <!-- signal-score: 0 | last-classified: 2026-06-29 | first-written: 2026-06-29 -->
