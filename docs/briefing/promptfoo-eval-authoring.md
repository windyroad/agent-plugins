# promptfoo Eval Authoring

Cross-session learnings about authoring `promptfooconfig.yaml` SKILL evals — Tier-A regex pitfalls, Nunjucks rendering, Tier-B llm-rubric routing. Split out of `governance-workflow.md` 2026-06-10 per Tier 3 budget rotation (P099 Branch B, split-by-subtopic).

## What You Need to Know

### promptfoo Tier-A regex uses JavaScript engine — Perl-style `(?im)` inline flags fail; negative-clause `not-icontains` false-fails on quoted-trailer-name (2026-06-08)

Authoring `promptfooconfig.yaml` Tier-A assertions: promptfoo evaluates regex via the JavaScript engine, not PCRE. Perl-style inline flag syntax `(?im)^...$` raises **"Invalid group"** parse error — use JS-style `/regex/im` flags (the `re` field in the YAML supports embedded flags) or simpler patterns. Worked failure (P324 Phase 3, 2026-06-08): `not-regex '(?im)^(?:[-*\d.\s]*)?JTBD-\d{3}\s*$'` parse-errored both targets. Separate adjacent pitfall: `not-icontains` Tier-A checks false-fail when a behaviourally-correct response **quotes the forbidden token to say it's dropped/forbidden** — e.g. asserting `not-icontains 'RISK_BYPASS: capture-deferred-readme'` against a response that says "the bypass trailer is dropped" trips the substring. The eval header comment block explicitly predicts this pattern (work-problems Phase 1 + review-problems Phase 2 reference both call it out) — route negative-clause semantics to **Tier-B llm-rubric**, keep Tier-A for distinctive positive tokens (verdict names, command names, file paths, classification tokens the LLM won't echo naturally). See `packages/itil/skills/capture-problem/eval/promptfooconfig.yaml` for the P324 Phase 3 worked example (6/6 GREEN after the Tier-A fix). <!-- signal-score: 0 | last-classified: 2026-06-08 | first-written: 2026-06-08 -->

### AGENT-prose evals (RFC-012 S1) run the agent via `claude -p --system-prompt "$(cat agent.md)"` (full replace, NOT `--append`) and cd to repo root; headless stdout is non-deterministic — Tier-A hard tokens (job IDs, verdict-template headers) flake ~50%, anchor Tier-A on the verdict-OUTCOME token only (2026-06-27)

The agent-prose twin of the SKILL evals (P324). Driver shape (`packages/jtbd/agents/eval/run-agent-eval.sh`): resolve repo root, `cd` there (so the agent's live `docs/jtbd/` reads + the `wr-jtbd-is-job-or-persona-unconfirmed` PATH shim's default root resolve), then `exec claude -p --system-prompt "$(cat ../agent.md)" "$@"`. Use **`--system-prompt` (full replace)** for an agent eval — its prose IS the entire instruction set — distinct from the SKILL evals' `--append-system-prompt` (ADR-075). **Key flake lesson:** under headless `claude -p`, the agent frequently writes its full verdict to the marker file (`/tmp/jtbd-verdict`) and emits only a TERSE summary to stdout — so Tier-A `icontains` on a specific job ID (`JTBD-006`) or the verdict-template header (`JTBD Review`) flaked ~50% across runs even though the verdict was behaviourally correct. **Fix:** anchor Tier-A on the verdict-OUTCOME token alone (`icontains: PASS`) — it survives terse + full forms — and route job-naming + all negative-clause semantics (the over-fire `[Unratified Dependency]` guard) to Tier-B `llm-rubric` (which tolerates phrasing variance). Verify GREEN over ≥3 consecutive runs before committing; single-sample pass^k robustness is RFC-012 S3's job, not S1's. Worked example: `packages/jtbd/agents/eval/promptfooconfig.yaml`. Adjacent: both governance oversight drains have CLOSED (0 unratified jtbd jobs/personas + 0 unratified ADRs as of 2026-06-27), so a live positive-fire `[Unratified Dependency]` fixture is no longer constructable against real docs — it needs a synthetic fixture corpus (the shim accepts an optional `[JTBD_DIR]` second arg). <!-- signal-score: 0 | last-classified: 2026-06-27 | first-written: 2026-06-27 -->

### Tarball-excluding a per-package eval dir: a `.npmignore` does NOT exclude a subdir of a `files`-allowlisted dir — use a `files`-field NEGATION entry (`!agents/eval/`) (2026-06-27)

When `package.json` has a `files` allowlist that includes a parent dir (e.g. `agents/`), a sibling `.npmignore` listing `agents/eval/` does NOT remove it from the tarball — npm includes the allowlisted dir wholesale. The working mechanism is a **negation entry in the `files` array itself**: add `"!agents/eval/"` after the allowlist entries (exactly as `packages/itil/package.json` does with `"!skills/*/eval/"`). The itil `.npmignore` that claims to be the exclusion mechanism is redundant/misleading — the `files`-negation is what actually fires. Verify with `npm pack --dry-run` from the package dir: no `agents/eval/` path should appear (P324 S1, ADR-075 Confirmation §1 tarball-exclusion). <!-- signal-score: 0 | last-classified: 2026-06-27 | first-written: 2026-06-27 -->

### promptfoo prompts + llm-rubric values are Nunjucks-rendered — literal `{{...}}` mustache examples must be wrapped in `{% raw %}...{% endraw %}` OR paraphrased; bare-`No.` answers + sentence-cap variants false-fail case-sensitive Tier-A regex (2026-06-09)

Two new Phase-5 pitfalls extending the Phase 3 entry. (1) **Nunjucks render error on `{{...}}` in prompt or rubric body**: promptfoo renders `vars.prompt` AND `llm-rubric.value` through Nunjucks before dispatch. A literal `{{project_name}}` in either surface raises `Template render error: expected name as lookup value, got .` (the `.`/`-` in token names trips Nunjucks's identifier parser). Wrap the offending fragments in `{% raw %}...{% endraw %}` blocks — works for both prompt and rubric — OR paraphrase to "mustache double-brace placeholders" so no literal `{{...}}` appears. Phase 5 worked example: P324 Phase 5 Step 4 token-substitution test ERRORed on first run; fixed by wrapping the prompt body + rubric body in `{% raw %}` blocks (still emits mustache `{{...}}` examples to the model). (2) **Tier-A regex is case-sensitive + no `(?i)` flag** (per Phase 3 finding) — sentence-cap variants like "No done marker." fail `no\s+done[\s\-]?marker` because of capital `N`. Enumerate case-variants explicitly via `[Nn]o` character class OR remove Tier-A and rely on Tier-B llm-rubric. (3) **Bare `No.` answers to framed Yes/No questions false-fail Tier-A regex expecting `does not X`** — when the prompt asks "Does it fire AskUserQuestion?", behaviourally-correct responses answer "**Does it fire AskUserQuestion?** No." — the "does not fire" substring never appears in the response (only in the question). Same negative-clause routing as Phase 3 — remove the Tier-A regex and rely on Tier-B llm-rubric per the documented capture-problem precedent.
