# promptfoo Eval Authoring

Cross-session learnings about authoring `promptfooconfig.yaml` SKILL evals ‚Äî Tier-A regex pitfalls, Nunjucks rendering, Tier-B llm-rubric routing. Split out of `governance-workflow.md` 2026-06-10 per Tier 3 budget rotation (P099 Branch B, split-by-subtopic).

## What You Need to Know

### promptfoo Tier-A regex uses JavaScript engine ‚Äî Perl-style `(?im)` inline flags fail; negative-clause `not-icontains` false-fails on quoted-trailer-name (2026-06-08)

Authoring `promptfooconfig.yaml` Tier-A assertions: promptfoo evaluates regex via the JavaScript engine, not PCRE. Perl-style inline flag syntax `(?im)^...$` raises **"Invalid group"** parse error ‚Äî use JS-style `/regex/im` flags (the `re` field in the YAML supports embedded flags) or simpler patterns. Worked failure (P324 Phase 3, 2026-06-08): `not-regex '(?im)^(?:[-*\d.\s]*)?JTBD-\d{3}\s*$'` parse-errored both targets. Separate adjacent pitfall: `not-icontains` Tier-A checks false-fail when a behaviourally-correct response **quotes the forbidden token to say it's dropped/forbidden** ‚Äî e.g. asserting `not-icontains 'RISK_BYPASS: capture-deferred-readme'` against a response that says "the bypass trailer is dropped" trips the substring. The eval header comment block explicitly predicts this pattern (work-problems Phase 1 + review-problems Phase 2 reference both call it out) ‚Äî route negative-clause semantics to **Tier-B llm-rubric**, keep Tier-A for distinctive positive tokens (verdict names, command names, file paths, classification tokens the LLM won't echo naturally). See `packages/itil/skills/capture-problem/eval/promptfooconfig.yaml` for the P324 Phase 3 worked example (6/6 GREEN after the Tier-A fix). <!-- signal-score: 0 | last-classified: 2026-06-08 | first-written: 2026-06-08 -->

### AGENT-prose evals (RFC-012 S1) run the agent via `claude -p --system-prompt "$(cat agent.md)"` (full replace, NOT `--append`) and cd to repo root; headless stdout is non-deterministic ‚Äî Tier-A hard tokens (job IDs, verdict-template headers) flake ~50%, anchor Tier-A on the verdict-OUTCOME token only (2026-06-27)

The agent-prose twin of the SKILL evals (P324). Driver shape (`packages/jtbd/agents/eval/run-agent-eval.sh`): resolve repo root, `cd` there (so the agent's live `docs/jtbd/` reads + the `wr-jtbd-is-job-or-persona-unconfirmed` PATH shim's default root resolve), then `exec claude -p --system-prompt "$(cat ../agent.md)" "$@"`. Use **`--system-prompt` (full replace)** for an agent eval ‚Äî its prose IS the entire instruction set ‚Äî distinct from the SKILL evals' `--append-system-prompt` (ADR-075). **Key flake lesson:** under headless `claude -p`, the agent frequently writes its full verdict to the marker file (`/tmp/jtbd-verdict`) and emits only a TERSE summary to stdout ‚Äî so Tier-A `icontains` on a specific job ID (`JTBD-006`) or the verdict-template header (`JTBD Review`) flaked ~50% across runs even though the verdict was behaviourally correct. **Fix:** anchor Tier-A on the verdict-OUTCOME token alone (`icontains: PASS`) ‚Äî it survives terse + full forms ‚Äî and route job-naming + all negative-clause semantics (the over-fire `[Unratified Dependency]` guard) to Tier-B `llm-rubric` (which tolerates phrasing variance). Verify GREEN over ‚â•3 consecutive runs before committing; single-sample pass^k robustness is RFC-012 S3's job, not S1's. Worked example: `packages/jtbd/agents/eval/promptfooconfig.yaml`. Adjacent: both governance oversight drains have CLOSED (0 unratified jtbd jobs/personas + 0 unratified ADRs as of 2026-06-27), so a live positive-fire `[Unratified Dependency]` fixture is no longer constructable against real docs ‚Äî it needs a synthetic fixture corpus (the shim accepts an optional `[JTBD_DIR]` second arg). <!-- signal-score: 0 | last-classified: 2026-06-27 | first-written: 2026-06-27 -->

### Tarball-excluding a per-package eval dir: a `.npmignore` does NOT exclude a subdir of a `files`-allowlisted dir ‚Äî use a `files`-field NEGATION entry (`!agents/eval/`) (2026-06-27)

When `package.json` has a `files` allowlist that includes a parent dir (e.g. `agents/`), a sibling `.npmignore` listing `agents/eval/` does NOT remove it from the tarball ‚Äî npm includes the allowlisted dir wholesale. The working mechanism is a **negation entry in the `files` array itself**: add `"!agents/eval/"` after the allowlist entries (exactly as `packages/itil/package.json` does with `"!skills/*/eval/"`). The itil `.npmignore` that claims to be the exclusion mechanism is redundant/misleading ‚Äî the `files`-negation is what actually fires. Verify with `npm pack --dry-run` from the package dir: no `agents/eval/` path should appear (P324 S1, ADR-075 Confirmation ¬ß1 tarball-exclusion). **Re-confirmed independently for `skills/*/eval/` on the FIRST risk-scorer eval (P381, 2026-06-28): adding `packages/risk-scorer/.npmignore` with `skills/*/eval/` changed nothing; moving `packages/itil/.npmignore` aside did NOT make itil's eval ship ‚Äî the decisive test. The fix was `"!skills/*/eval/"` in `packages/risk-scorer/package.json` `files`. CAUTION: the `wr-architect:agent` gate itself initially mis-attributed the mechanism to the `.npmignore` ‚Äî verify empirically with `npm pack --dry-run`, never trust the `.npmignore` presence (or the architect's reasoning about it) as proof. When adding the FIRST eval under a plugin with no `.npmignore`, you MUST add the `files`-negation or the eval ships.** <!-- signal-score: 2 | last-classified: 2026-06-28 | first-written: 2026-06-27 -->

### promptfoo prompts + llm-rubric values are Nunjucks-rendered ‚Äî literal `{{...}}` mustache examples must be wrapped in `{% raw %}...{% endraw %}` OR paraphrased; bare-`No.` answers + sentence-cap variants false-fail case-sensitive Tier-A regex (2026-06-09)

Two new Phase-5 pitfalls extending the Phase 3 entry. (1) **Nunjucks render error on `{{...}}` in prompt or rubric body**: promptfoo renders `vars.prompt` AND `llm-rubric.value` through Nunjucks before dispatch. A literal `{{project_name}}` in either surface raises `Template render error: expected name as lookup value, got .` (the `.`/`-` in token names trips Nunjucks's identifier parser). Wrap the offending fragments in `{% raw %}...{% endraw %}` blocks ‚Äî works for both prompt and rubric ‚Äî OR paraphrase to "mustache double-brace placeholders" so no literal `{{...}}` appears. Phase 5 worked example: P324 Phase 5 Step 4 token-substitution test ERRORed on first run; fixed by wrapping the prompt body + rubric body in `{% raw %}` blocks (still emits mustache `{{...}}` examples to the model). (2) **Tier-A regex is case-sensitive + no `(?i)` flag** (per Phase 3 finding) ‚Äî sentence-cap variants like "No done marker." fail `no\s+done[\s\-]?marker` because of capital `N`. Enumerate case-variants explicitly via `[Nn]o` character class OR remove Tier-A and rely on Tier-B llm-rubric. (3) **Bare `No.` answers to framed Yes/No questions false-fail Tier-A regex expecting `does not X`** ‚Äî when the prompt asks "Does it fire AskUserQuestion?", behaviourally-correct responses answer "**Does it fire AskUserQuestion?** No." ‚Äî the "does not fire" substring never appears in the response (only in the question). Same negative-clause routing as Phase 3 ‚Äî remove the Tier-A regex and rely on Tier-B llm-rubric per the documented capture-problem precedent. See `packages/itil/skills/scaffold-intake/eval/promptfooconfig.yaml` for the P324 Phase 5 worked example (7/7 GREEN after the three fixes). <!-- signal-score: 0 | last-classified: 2026-06-09 | first-written: 2026-06-09 -->

### A flaky SKILL eval suite usually has MULTIPLE brittle cases, not one ‚Äî and not every flake is Tier-A: the Tier-B rubric can CORRECTLY fail a genuine SKILL-prose self-contradiction (2026-06-27)

Two lessons from P393 (run-retro eval, 6/7-intermittent). (1) **Don't whack-a-mole one case ‚Äî audit the whole suite.** When a suite reports "1 failed" but a re-run is green, it is FLAKY, not deterministically red. Each case carrying multi-part / negative-clause Tier-A regexes flakes ~1-in-3 independently, so for a 5-such-case suite P(all green) ‚âà (2/3)^5 ‚âà 0.13 ‚Äî you will keep hitting a *different* failing case each run. Fix once: demote the semantic / negative-clause Tier-A regexes across EVERY case to the (already comprehensive) Tier-B `llm-rubric`, keeping Tier-A only for distinctive paraphrase-proof positive anchors (numeric thresholds, command names, citation / classification tokens). (2) **When demoting Tier-A doesn't stabilise a case, read WHICH assert fails before assuming brittleness.** In P393 the residual flaky case (Step 1.5) had both retained Tier-A anchors PASSING and the **Tier-B llm-rubric FAILING** ‚Äî because the model gave a genuinely *wrong* answer ~1-in-3. Root cause was a real SKILL.md self-contradiction (Step 1.5 said `<=-3` deletes fire a batched `AskUserQuestion`; Step 3 + CLAUDE.md P132 + ADR-044 said briefing removals are silent). The model nondeterministically followed the stale prose. The eval was working as designed ‚Äî it caught a SKILL defect. **Fix the SKILL prose, never weaken the rubric to accept the wrong answer.** Use `--output <json>` + parse `gradingResult.componentResults` to see per-assert pass/fail; a failing Tier-B with passing Tier-A is the tell that the SKILL-under-test is wrong, not the eval. Verify GREEN over ‚â•3 consecutive runs (LLM non-determinism) before calling it stable. <!-- signal-score: 0 | last-classified: 2026-06-27 | first-written: 2026-06-27 -->

### For a SKILL-prose behavioural test, author a promptfoo eval case ‚Äî NOT a structural bats fixture: ADR-052's 2026-06-09 amendment DELETED the structural escape hatch (no `tdd-review: structural-permitted` justification works any more), even when mirroring the existing P146/P232 grep fixtures (2026-06-28)

When a SKILL.md prose change needs a behavioural second-source, reach for a **promptfoo eval case** (add it to the skill's `eval/promptfooconfig.yaml`), NOT a `grep`-on-SKILL.md bats fixture ‚Äî even though many such structural fixtures already exist in-tree (`work-problems-step-5-bats-polling-discipline.bats` etc.). Those pre-existing fixtures are **tracked debt** (P290 Phase 2 / blocked on P324), recorded as a known state-of-violation ‚Äî NOT a pattern to copy. ADR-052's **2026-06-09 amendment (Option 1A, behavioural-only, no escape hatch)** removed the former "Surface 2" / structural-permitted carve-out in its entirety: structural assertions on prose-document content (`SKILL.md`, `agent.md`, ADR bodies) are "not permitted under any justification". ADR-037 (which those fixtures cite as their "Permitted Exception" authority) is **superseded by ADR-052**. The `wr-architect:agent` edit-gate returns ISSUES FOUND on a net-new structural SKILL-prose grep fixture and withholds the marker (it only sets the marker on a PASS verdict ‚Äî `architect-mark-reviewed.sh`), blocking the edit. The clean path: a promptfoo case that exercises the skill-shape and asserts on the response (Tier-A paraphrase-proof anchors + Tier-B llm-rubric) ‚Äî which ALSO discharges the R009 in-source floor for the prose change per ADR-061 Rule 4.
