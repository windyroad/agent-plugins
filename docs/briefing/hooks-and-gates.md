# Hooks and Gates

Learnings about PreToolUse/PostToolUse gate behaviour, markers, and expiry.

## What You Need to Know

- **Pre-edit gate review catch-22 — the gate agent withholds PASS because "edits aren't applied yet", but the gate blocks the edits.** The architect/JTBD edit-gates fire BEFORE the Write, yet the review agents reason as if reviewing already-applied changes: handed a pre-edit plan they return ISSUES FOUND / withhold PASS solely because "the change isn't on disk / the old state is still live" — exactly what a pre-edit gate guarantees. Re-run the agent with an explicit preamble: *"This is a PRE-EDIT gate; the proposed change is NOT yet on disk by design — classify alignment of the PROPOSAL, do not treat not-yet-applied as an issue."* That re-run returns PASS. Hit on BOTH architect (RFC-006 write) and jtbd (after a same-session JTBD-policy edit invalidated the marker) in the 2026-05-26 ADR-070/071 session. Distinct from P181 (verdict-grep hook) and P303 (hash-drift deadlock). See P313. <!-- signal-score: 0 | last-classified: 2026-05-26 | first-written: 2026-05-26 -->

- **Multi-decision-file changes DEADLOCK the architect edit-gate** (P303 open) — verdict-grep reads "ISSUES FOUND" as FAIL, the gate re-locks on every `docs/decisions/` hash drift, post-approval re-reviews fail (work not on disk yet). Recover after a genuine approval (user-authorised): assert `/tmp/architect-reviewed-$SID` + `rm` the `.hash` sibling. The same verdict-grep false-FAIL (P181/P217) fires on a SINGLE-file edit when an APPROVED re-review's prose merely contains the substring `ISSUES FOUND` — same marker-assert recovery. (P261 hit this 2026-05-26.) <!-- signal-score: 2 | last-classified: 2026-05-26 | first-written: 2026-05-25 -->
- **A commit that removes/narrows a load-bearing currency hook is blocked by the cached OLD hook, and `BYPASS_*=1` can't reach it (P173 verifying) — land it from an EXTERNAL terminal.** The cached hook lags source so the superseded rule still fires, and the PreToolUse gate only intercepts in-session commits. Write the message to a repo-local real path (`.git/<name>.txt`); the session's `/tmp` is sandboxed and invisible to the user's terminal. <!-- signal-score: 0 | last-classified: 2026-05-25 | first-written: 2026-05-25 -->
- **Within-appetite bypass markers (`reducing-commit` / `reducing-push` / `reducing-release`) PERSIST across multiple commits within `RISK_TTL` window when the pipeline-state hash matches what was scored** (P192 verifying, 2026-06-08). Mirrors the in-repo `clean`-marker persist-until-drift precedent. Drift (tree changed beyond P054 tree-stable + doc-excluded) OR TTL expiry consumes the marker and forces a fresh `wr-risk-scorer:pipeline` rescore. CONTRAST with `incident-release` / `ci-bypass-${ACTION}`, which remain DELIBERATELY single-use (one-time overrides, regression-guarded by `reducing-marker-persistence.bats` test 10). The architectural distinction: within-appetite verdicts persist-until-drift; deliberate overrides stay single-use. Eliminates the per-commit rescore round-trip that drove ≥3 rescores per multi-commit session historically. <!-- signal-score: 0 | last-classified: 2026-06-08 | first-written: 2026-06-08 -->

## What Will Surprise You

- **The external-comms gate hashes the reviewed `<draft>` body and compares it against the `gh issue comment --body` argument — backticks break the match.** A draft containing markdown backticks must be backslash-escaped inside the double-quoted `--body "..."` shell argument, and the gate's marker key derivation sees the escaped text, so a PASS review of the backticked draft never unlocks the post (two wasted ~21K-token review round-trips observed 2026-06-11, AFK iter 3, P228 upstream comment). Write outbound `gh` bodies WITHOUT backticks (plain names instead of code spans), then review + post the byte-identical text. Also: the risk evaluator and voice-tone evaluator each need their own structured review (`SURFACE: gh-issue-comment` first line + `<draft>...</draft>` wrapper) — they unlock serially, not together. <!-- signal-score: 0 | last-classified: 2026-06-11 | first-written: 2026-06-11 -->
- **Copying an existing hook as a template silently propagates its latent BSD-awk portability bugs — and only the deny path exposes them.** The architect `architect-compendium-refresh-discipline.sh` (now retired by ADR-078 Story D) inlined a leading-`git commit` token check with `awk '/^git[[:space:]]+commit\b/'`. The `\b` word-boundary is a GNU-awk extension — BSD awk on macOS does NOT match it, so the check returns false, falls through `|| exit 0`, and the hook silently never fires its deny. Permit-path bats stay green either way (they expect exit 0); only a deny-path fixture caught it (P337 iter 10, 2026-06-16). Portable form: `commit([[:space:]]|$)`. Better still, the itil P165 sibling (`itil-readme-refresh-discipline.sh`) does NOT inline awk at all — it delegates to a shared, tested `lib/command-detect.sh::command_invokes_git_commit` helper (P268 family: tickets 268/272/274/275). Architect hooks should use the same shared helper rather than re-inlining commit detection. Same BSD/GNU awk-divergence class as P334/P328. <!-- signal-score: 2 | last-classified: 2026-06-16 | first-written: 2026-06-16 -->
