- **The I13 fix-time RFC-trace gate (`wr-itil-check-fix-rfc-trace`) auto-create directive has NO wire-existing-vehicle branch yet (P371 open).** When you propose a fix on a Known Error, the gate fires `no-rfc-trace: P<NNN>` and the manage-problem directive says "auto-create a skeleton RFC via capture-rfc". But if an existing RFC IS the ticket's fix vehicle and simply hasn't wired the problem into its `problems:` array (the rework/follow-on case — the RFC names the original driver, not this ticket), following the directive literally creates a REDUNDANT duplicate RFC. Before trusting auto-create, run the manual pre-check: enumerate every `docs/rfcs/RFC-*.md` `problems:` array AND `grep` bodies for the fix surface; if a vehicle exists → wire the trace edge into its `problems:` array + refresh the ticket's derived `## RFCs` section (`wr-itil-update-problem-rfcs-section`), do NOT auto-create. Auto-create is correct ONLY when no vehicle exists (ADR-073). Worked clean auto-create dogfood: P361 → RFC-026 (RFC-005 B8, 2026-06-18). ## What Will Surprise You

- **The external-comms gate hashes the reviewed `<draft>` body and compares it against the `gh issue comment --body` argument — backticks break the match.** A draft containing markdown backticks must be backslash-escaped inside the double-quoted `--body "..."` shell argument, and the gate's marker key derivation sees the escaped text, so a PASS review of the backticked draft never unlocks the post (two wasted ~21K-token review round-trips observed 2026-06-11, AFK iter 3, P228 upstream comment). Write outbound `gh` bodies WITHOUT backticks (plain names instead of code spans), then review + post the byte-identical text. Also: the risk evaluator and voice-tone evaluator each need their own structured review (`SURFACE: gh-issue-comment` first line + `<draft>...</draft>` wrapper) — they unlock serially, not together. <!-- signal-score: 0 | last-classified: 2026-06-11 | first-written: 2026-06-11 -->
- **Copying an existing hook as a template silently propagates its latent BSD-awk portability bugs — and only the deny path exposes them.** The architect `architect-compendium-refresh-discipline.sh` (now retired by ADR-078 Story D) inlined a leading-`git commit` token check with `awk '/^git[[:space:]]+commit\b/'`. The `\b` word-boundary is a GNU-awk extension — BSD awk on macOS does NOT match it, so the check returns false, falls through `|| exit 0`, and the hook silently never fires its deny. Permit-path bats stay green either way (they expect exit 0); only a deny-path fixture caught it (P337 iter 10, 2026-06-16). Portable form: `commit([[:space:]]|$)`. Better still, the itil P165 sibling (`itil-readme-refresh-discipline.sh`) does NOT inline awk at all — it delegates to a shared, tested `lib/command-detect.sh::command_invokes_git_commit` helper (P268 family: tickets 268/272/274/275). Architect hooks should use the same shared helper rather than re-inlining commit detection. Same BSD/GNU awk-divergence class as P334/P328. <!-- signal-score: 2 | last-classified: 2026-06-16 | first-written: 2026-06-16 -->
