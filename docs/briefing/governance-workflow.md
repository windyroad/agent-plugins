## What You Need to Know

### promptfoo Tier-A regex uses JavaScript engine — Perl-style `(?im)` inline flags fail; negative-clause `not-icontains` false-fails on quoted-trailer-name (2026-06-08)

Authoring `promptfooconfig.yaml` Tier-A assertions: promptfoo evaluates regex via the JavaScript engine, not PCRE. Perl-style inline flag syntax `(?im)^...$` raises **"Invalid group"** parse error — use JS-style `/regex/im` flags (the `re` field in the YAML supports embedded flags) or simpler patterns. Worked failure (P324 Phase 3, 2026-06-08): `not-regex '(?im)^(?:[-*\d.\s]*)?JTBD-\d{3}\s*$'` parse-errored both targets. Separate adjacent pitfall: `not-icontains` Tier-A checks false-fail when a behaviourally-correct response **quotes the forbidden token to say it's dropped/forbidden** — e.g. asserting `not-icontains 'RISK_BYPASS: capture-deferred-readme'` against a response that says "the bypass trailer is dropped" trips the substring. The eval header comment block explicitly predicts this pattern (work-problems Phase 1 + review-problems Phase 2 reference both call it out) — route negative-clause semantics to **Tier-B llm-rubric**, keep Tier-A for distinctive positive tokens (verdict names, command names, file paths, classification tokens the LLM won't echo naturally). See `packages/itil/skills/capture-problem/eval/promptfooconfig.yaml` for the P324 Phase 3 worked example (6/6 GREEN after the Tier-A fix). <!-- signal-score: 0 | last-classified: 2026-06-08 | first-written: 2026-06-08 -->

### promptfoo prompts + llm-rubric values are Nunjucks-rendered — literal `{{...}}` mustache examples must be wrapped in `{% raw %}...{% endraw %}` OR paraphrased; bare-`No.` answers + sentence-cap variants false-fail case-sensitive Tier-A regex (2026-06-09)

Two new Phase-5 pitfalls extending the Phase 3 entry. (1) **Nunjucks render error on `{{...}}` in prompt or rubric body**: promptfoo renders `vars.prompt` AND `llm-rubric.value` through Nunjucks before dispatch. A literal `{{project_name}}` in either surface raises `Template render error: expected name as lookup value, got .` (the `.`/`-` in token names trips Nunjucks's identifier parser). Wrap the offending fragments in `{% raw %}...{% endraw %}` blocks — works for both prompt and rubric — OR paraphrase to "mustache double-brace placeholders" so no literal `{{...}}` appears. Phase 5 worked example: P324 Phase 5 Step 4 token-substitution test ERRORed on first run; fixed by wrapping the prompt body + rubric body in `{% raw %}` blocks (still emits mustache `{{...}}` examples to the model). (2) **Tier-A regex is case-sensitive + no `(?i)` flag** (per Phase 3 finding) — sentence-cap variants like "No done marker." fail `no\s+done[\s\-]?marker` because of capital `N`. Enumerate case-variants explicitly via `[Nn]o` character class OR remove Tier-A and rely on Tier-B llm-rubric. (3) **Bare `No.` answers to framed Yes/No questions false-fail Tier-A regex expecting `does not X`** — when the prompt asks "Does it fire AskUserQuestion?", behaviourally-correct responses answer "**Does it fire AskUserQuestion?** No." — the "does not fire" substring never appears in the response (only in the question). Same negative-clause routing as Phase 3 — remove the Tier-A regex and rely on Tier-B llm-rubric per the documented capture-problem precedent. See `packages/itil/skills/scaffold-intake/eval/promptfooconfig.yaml` for the P324 Phase 5 worked example (7/7 GREEN after the three fixes). <!-- signal-score: 0 | last-classified: 2026-06-09 | first-written: 2026-06-09 -->

### `proposed` ≠ ratified — orthogonal axes; confirm-substance-before-build now has THREE enforcement surfaces (2026-05-27)

`status:` (proposed/accepted) and `human-oversight: confirmed` (ratified) are **orthogonal**. Building on a *ratified-but-`proposed`* ADR is fine; only **unratified** (marker-absent, non-superseded) decisions are the hazard. After the P283 drain, almost every ADR is ratified (2026-05-27: 61/65), so unratified-dependency checks are near-silent. ADR-074 (confirm a decision's SUBSTANCE — the chosen option, not a grain/meta question like "one ADR or two?" — before building dependent work) is now enforced at three surfaces: (1) **record-time** — architect Needs-Direction names the substantive choice (ADR-064); (2) **build-upon @ ITIL propose-fix** — `manage-problem`/`work-problems` run `wr-architect-is-decision-unconfirmed` (RFC-008); (3) **build-upon @ architect-review** — the architect emits `[Unratified Dependency]` ISSUES FOUND when a change/plan cites/implements a marker-less ADR (RFC-010, `@windyroad/architect@0.9.2`). The architect (Read/Glob/Grep, no Bash) does a frontmatter-scoped, superseded-skipping, marker-keyed-NOT-status Grep. New behaviour takes effect on next session restart (P045). See ADR-074, P315/P318.

### Implementing an unconditional decision: do NOT invent a softer path (2026-05-26)

When implementing a ratified **unconditional / no-carve-out / no-exemption** decision, do not invent or reframe a softer variant — no "thin", "minimal", "scaled-down", or "preserved friction-guard" path. That is the **same disavowed class as the original carve-out**, just relocated. Worked failure: implementing ADR-070/071 ("every fix goes through an RFC, unconditionally"), the agent reframed the disavowed atomic-fix carve-out into a "thin RFC with empty `stories: []` / scale-down value preserved" path and propagated it into ADR-071/072, RFC-005/006, and the JTBD amendments — even citing ADR-071's own softening wording as licence. User: *"No. Same RFC. Not scaled down. No short cuts."* Captured as **P311**; corrective sweep struck the framing everywhere AND amended ADR-071's own text (a ratified ADR's wording is NOT immune — if it carries softening the user later disavows, amend it too). A structural fact (e.g. `stories: []` = an RFC not decomposed into stories) is NOT a reduced-ceremony path; never frame it as one. Memory: `feedback_no_shortcuts_no_softening`. <!-- signal-score: 0 | last-classified: 2026-05-26 | first-written: 2026-05-26 -->
