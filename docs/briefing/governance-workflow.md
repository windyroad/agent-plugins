## What You Need to Know

### WSJF is computed ONLY at README render time — transitions silently leave the cache stale (2026-06-28)

Problem tickets carry `**Priority**:` (Impact×Likelihood) and `**Effort**:` lines but NO `**WSJF**:` line — WSJF is derived at render time in `docs/problems/README.md` by `/wr-itil:review-problems` Step 5. So a status transition (`/wr-itil:transition-problem`, work-iter O→KE) that changes the WSJF multiplier (Open 1.0 → Known Error 2.0) does NOT recompute the README's WSJF column. The ranking silently shows the Open-era value until the next `review-problems` re-render. Witnessed 2026-06-28: 11 KE-cluster tickets transitioned O→KE on 2026-06-16 still showed WSJF 1.5 (should be 3.0); P390 showed 6.0 (should be 12.0 — Sev-12 KE); recent captures showed 0.75 placeholders. `reconcile-readme.sh` does NOT catch this — it checks section *membership*, not WSJF *arithmetic/sort order*. So the WSJF table can be badly mis-sorted (recent captures prepended above higher-WSJF rows) and reconcile still exits 0. Implication: trust the README WSJF ranking only right after a `review-problems` pass; a mid-cycle backlog will lag reality. Candidate fix space (P389-adjacent): have the transition skills refresh WSJF in the same commit, or add a render-correctness check to reconcile. <!-- signal-score: 2 | last-classified: 2026-06-28 | first-written: 2026-06-28 -->

### "No schema change" ≠ "no ADR" — a direction-extension still needs an amendment (2026-06-22)

When a fix extends a governance contract by adding a *direction* / *consumer* / *branch* — even when it introduces NO new data schema and reuses an existing field — the architect will require recording it as an **amendment to the owning ADR**, not "no ADR at all". Worked example (P363 inbound-origin verdict dispatch): consuming the already-existing `**Origin**: inbound-reported (#NN)` field (ADR-076) to add an inbound verdict leg to `/wr-itil:update-upstream` was *not* a new schema, but it added a new *direction* to ADR-024's wholly-outbound contract — so it needed an ADR-024 amendment (same mechanism as that ADR's P080/P362/Phase-2 amendments), NOT a standalone ADR and NOT zero ADR. The orchestrator's "follows existing schema → no ADR required" framing was half-right (no standalone ADR) and half-wrong (an amendment IS required). Budget the architect round-trip: pre-draft the amendment entry and re-run the architect to get a clean APPROVED marker before editing. Also: amending an existing ADR fires the architect-compendium PostToolUse hook — verify `grep -ac "### ADR-" docs/decisions/README.md` against HEAD after the edit (memory `feedback_compendium_hook_truncates_readme_on_adr_edit`). And per P357/ADR-066, user-ratification of the *mechanism* (e.g. "do option (b)") does NOT ratify the implementation substance (template wording / resolution semantics / new tokens) — queue that to `outstanding_questions` under AFK. <!-- signal-score: 0 | last-classified: 2026-06-22 | first-written: 2026-06-22 -->

### `proposed` ≠ ratified — orthogonal axes; confirm-substance-before-build now has THREE enforcement surfaces (2026-05-27)

`status:` (proposed/accepted) and `human-oversight: confirmed` (ratified) are **orthogonal**. Building on a *ratified-but-`proposed`* ADR is fine; only **unratified** (marker-absent, non-superseded) decisions are the hazard. After the P283 drain, almost every ADR is ratified (2026-05-27: 61/65), so unratified-dependency checks are near-silent. ADR-074 (confirm a decision's SUBSTANCE — the chosen option, not a grain/meta question like "one ADR or two?" — before building dependent work) is now enforced at three surfaces: (1) **record-time** — architect Needs-Direction names the substantive choice (ADR-064); (2) **build-upon @ ITIL propose-fix** — `manage-problem`/`work-problems` run `wr-architect-is-decision-unconfirmed` (RFC-008); (3) **build-upon @ architect-review** — the architect emits `[Unratified Dependency]` ISSUES FOUND when a change/plan cites/implements a marker-less ADR (RFC-010, `@windyroad/architect@0.9.2`). The architect (Read/Glob/Grep, no Bash) does a frontmatter-scoped, superseded-skipping, marker-keyed-NOT-status Grep. New behaviour takes effect on next session restart (P045). See ADR-074, P315/P318.
