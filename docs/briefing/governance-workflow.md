# Governance Workflow

Cross-session learnings about ADRs, architect and JTBD reviews, risk scoring, voice-tone, and the session-wide unifying patterns (progressive disclosure, canonical+sync, SKILL+REFERENCE).

## What You Need to Know

- **Risk appetite is Low (4)**. Changes scoring Medium (5+) need explicit acknowledgement. See `RISK-POLICY.md`.
- **All ADRs in `docs/decisions/` are still `.proposed.md`** — none ratified. Amendments are cheap: prefer amending over superseding. When a P-problem intersects a proposed ADR, revise the ADR rather than adding a compatibility clause.
- **ADR-002 plugin dependency graph lists `@windyroad/tdd` as standalone.** Cross-plugin features (e.g. P018 JTBD traceability, P017 shared concern-splitting helper) must update the graph explicitly; don't silently add deps.
- **Same-day ADRs may conflict — cross-reference before implementing a ticket direction.** A direction decision added to a ticket today and a new ADR accepted the same day on the same files may carry different execution models (e.g. synchronous Step-0 delegation vs asynchronous background-subagent invocation). The architect catches this, but the wasted iteration can be avoided by greping `docs/decisions/*.proposed.md` for files dated the same day as the direction and spot-checking the overlap before writing implementation code. Observed 2026-04-20 AFK iter 2: P014's 2026-04-20 direction and ADR-027's 2026-04-20 acceptance collided on manage-problem/run-retro/create-adr SKILL.md files; architect flagged the conflict and the iteration skipped with findings. See P014.
- **Architect reviews routinely surface 1-3 "bundle these in the same commit" follow-up edits.** A commit that touches ADR-X may require a one-line edit to ADR-Y (explicit cross-reference), or an adjacent skill's SKILL.md carve-out, or a missing rationale section. Factor this into effort estimates — "L" is usually "L + architect-added bundling" at the upper bound, not the midpoint. Observed repeatedly 2026-04-20 AFK iters 3/4/5. This is the architect working as designed; don't resist it — bundle.
- **Progressive disclosure is the session-wide unifying solution pattern for context-budget problems** per ADR-038 (2026-04-22). Less info upfront + explicit affordances (agent pointers, REFERENCE.md paths, turn-1 references) so consumers expand on demand rather than having context re-injected. Companion axes: trimming, consolidation, gating, scope-awareness. Applied first to UserPromptSubmit hooks (P095 shipped); candidates at PreToolUse/PostToolUse (P096), SKILL.md runtime size (P097), user/project-owned files (P098). Anchor ticket P091.
- **Canonical+sync is the pattern for cross-plugin shared code** (ADR-017 for `.mjs`, ADR-028/ADR-038 extend to `hooks/` and `hooks/lib/`). Three-step scaffold: (1) canonical at `packages/shared/{,hooks/lib/}<name>`, (2) per-plugin synced copies, (3) `scripts/sync-<name>.sh` with `--check` mode + npm `sync:<name>` / `check:<name>` scripts + CI drift step + `packages/shared/test/sync-<name>.bats`. Each plugin sources its LOCAL copy, never the canonical — preserves ADR-002 installable-independently. Precedents: install-utils.mjs (P026), session-marker.sh (P095).
- **SKILL.md + sibling REFERENCE.md is the progressive-disclosure pattern for skill bodies** (ADR-038 applied to skill runtime size — formal ADR expected on P097 when the pattern is generalised across the windyroad suite). Shape: `SKILL.md` carries runtime steps, bash snippets, report templates, contract invariants, and explicit pointers to `REFERENCE.md` anchors where deeper rationale is needed; `REFERENCE.md` carries BRIEFING references, per-ADR rationale, edge cases, scope exclusions, and the "not in scope" list. Consumer loads REFERENCE on demand, not eagerly. Reference implementation: source-of-truth at `scripts/repo-local-skills/install-updates/{SKILL,REFERENCE}.md` (relocated 2026-04-28 per P139 — `.claude/skills/install-updates/` now carries relative symlinks per ADR-030's Symlink Contract; original P098 2026-04-22 split-pattern unchanged). Before generalising across windyroad plugins, P097 should author a sibling ADR naming a byte budget per SKILL.md (discoverable via `performance-budget-*` glob per ADR-023).
- **Investigated outcome with architect-defaults inline is the AFK-iter shape when investigation surfaces ≥ 2 design questions** — append `## Architect findings`, `## JTBD findings`, and `## Outstanding Design Questions` sections to the ticket; per Outstanding Design Question, give an architect-preferred default plus 2-3 alternatives the user can answer in one batch on return. Avoids the P061 cascading `AskUserQuestion`-batch anti-pattern (ADR-032:203) AND respects the JTBD-006 trust boundary ("problems requiring my judgment are queued for my return, not guessed at"). Distinct from "bundle architect's same-commit follow-ups" (which is in-iter scope expansion) — this pattern packages the deferred design choices into a single batched-answer artefact on the ticket itself. Observed 2026-04-22 AFK iter 2 on P105: architect raised three design questions (interactive shape / persistence format / delete guard rail) with mild-conviction defaults; the iter committed findings + Outstanding Design Questions and exited investigated rather than guessing the answers.
- **User direction: reject BUFD ranked-iteration.** When the scorer suggests remediations, the agent reads them and decides what to do incrementally — like any other workflow decision. There is no requirement to rank all suggestions upfront or iterate through them in order. Adding structured `action_class` columns, known-class tables, or pre-built executors is over-engineering. The agent reads prose and decides. Observed 2026-04-23: ADR-042 was simplified three times — first from closed enumeration to open vocabulary, then from open vocabulary with expressibility check to "agent decides", then from ranked-iteration to incremental decision-making. User verbatim: *"this should NOT be a BUFD thing where it iterates over the suggestions. It should just get the suggestions and decide what to do, incrementally like we do for everything else."*
- **Gate approvals validate the proposed change, not the underlying hypothesis.** Architect + JTBD + risk-scorer subagents review a concrete edit against documented rules and decisions; they cannot validate that the edit's theoretical *cause-effect link* is correct. A passing gate only proves "this edit does not violate recorded rules". Observed 2026-04-24 on P113: the hypothesis ("undeclared `Skill, Agent` in `allowed-tools` causes Claude Code TUI to drop the skill from autocomplete") passed architect review (correct ADR-015 alignment), JTBD review (serves JTBD-001 + JTBD-301), and risk-scorer (2/25 at all layers). Fix shipped to npm. User re-tested — autocomplete still broken. The actual cause was a stale worktree install entry, unrelated to the plugin's frontmatter. Cost: one release cycle + one reinstall + one session restart. **Takeaway**: before committing to a hypothesis whose verification path requires `release + reinstall + restart + re-test`, run the cheap diagnostic tools first (`claude plugin list` for plugin-autocomplete issues, `git log @{push}..HEAD` for CI-regression issues, `claude plugin marketplace list` for marketplace-resolution issues). A 30-second diagnostic beats a 10-minute release cycle that ships a wrong hypothesis. This is the investigation-ordering analogue of the JTBD-201 "evidence-before-mitigation" principle.
- **The auto-create-on-missing-doc pattern is the established contract for doc-bearing review gates.** When `docs/VOICE-AND-TONE.md`, `docs/STYLE-GUIDE.md`, or `docs/jtbd/` is missing in a project, the corresponding `*-enforce-edit.sh` PreToolUse hook calls `review_gate_deny` with a message directing to `/wr-<plugin>:update-guide`. The gate never silently disables itself — it blocks the edit and routes to the doc-creation skill so the rubric exists before reviews can fire. Architect gate is structurally similar (inactive only when `docs/decisions/` is entirely absent; once present, blocks until reviewed). New plugins that introduce a doc-bearing gate should follow this contract — block + route to the create-doc skill, never silently degrade to "review skipped because doc missing". Confirmed 2026-04-25 reading `packages/{voice-tone,style-guide,jtbd}/hooks/*-enforce-edit.sh` lines 56-60 directly.
- **Post-landing ADR cleanup is a recurring shape.** When an ADR lands in 2-4 quick commits walking back its own first draft, the **Rules** section gets revised cleanly but **Decision Drivers / Decision Outcome / Consequences** often carry stale references to constructs that were stripped. Budget a follow-on cleanup pass reconciling rationale prose with the current Rules — and before proposing the amendment, reread any user direction dated AFTER the ADR's initial landing commit, since the framing may have evolved (e.g. "innovation vs standardisation as a spectrum" was a post-landing refinement to ADR-042's JTBD-Gap disposition). Observed 2026-04-24 reviewing ADR-042: commits 60ed530 → 425266e → 64f6d3f → 6d5f220 stripped BUFD ranked-iteration, `action_class`, the known-class table, and novel-class parsing from Rule 2a, but "expressibility check" / "bounded parsing" / "stable remediation-class vocabulary" phrases survived in Decision Drivers + Decision Outcome + Good consequence #7; a subsequent amendment (ce5cfb6) added an Innovation Window section and stripped the residue. **Also**: an ADR's acknowledged JTBD Gap framed as "follow-up" may actually be deliberate policy once the user's direction clarifies — reframe rather than backfilling a premature JTBD.
- **Meta-recursive bootstrap is acceptable when introducing framework primitives.** When codifying a new framework primitive (RFC, lifecycle stage, ADR class), the framework's own first artefact may have to live under the existing structure pending Phase 1 implementation — the RFC framework can't exist without the decision-capture that introduces it. Acknowledge the recursion explicitly (e.g. P170's *"Implementation Tasks (workaround section — until RFC framework arrives)"* with a forward pointer, or ADR-060's Phase 1 scope clause naming P168 as the first retrospective RFC). The user-correction signal that surfaces this conflation tends to be *"are these X or are they Y?"* — soft-challenge phrasing to a flat list that's mixing two task classes. Apply the structural split inline; don't paper over the recursion. Observed 2026-05-04 / 2026-05-05 on P170 + ADR-060 RFC framework introduction.
- **`docs/risks/` is the persistent catalog of per-action risks — NOT a separate aggregation from `.risk-reports/`.** Per RISK-POLICY.md commit `9e339d0` `## Risk Catalog` section: per-action assessments READ the catalog, filter to risks applicable to THIS action, assess controls FOR THIS ACTION against the **same 4/Low appetite**, and append newly-conceived risk classes back to the catalog. There is no per-action vs lifetime aggregation distinction; same appetite uniformly. A catalog residual above appetite is a **real signal** that baseline controls are insufficient for typical actions — either add controls (drop baseline residual) or accept that per-action assessments must add specific controls each time. The waste this design eliminates: re-deriving the same ~327 risk titles across 181 reports per gap analysis. The agent today implements neither the consume-catalog nor the bootstrap-from-empty workflow — see P168 for the substantive design ticket. Two-correction-round cost on 2026-05-04 (P167 misreading) before the simple catalog model surfaced; keep the framing simple on first read. ADR-042's "never release above appetite" invariant remains in force for per-action pipeline scoring.
- **Substantive ADR amendments after architect+JTBD reviews trigger a 3-pass review cycle on accept.** When the architect or JTBD review surfaces ≥ 5 amendments on a `.proposed.md` ADR, applying those amendments invalidates the prior reviews — the JTBD edit-gate detects "policy file changed since last review" on the rename + status flip and re-fires; the architect can also re-fire if the substantial amendments changed any cited driver. Budget for **3 review passes** when accepting a substantively-amended ADR: (1) initial architect review, (2) initial JTBD review, (3) re-review on accept (one or both gates re-fire). Each pass is 30-90s of subagent time. Observed 2026-05-05 on ADR-060 accept after 14 architect + 8 JTBD amendments — the JTBD policy-changed re-fire was correct (JTBD-001 + JTBD-101 had been edited as part of the amendments). The re-review found no new issues; the cycle is structural, not redundant, because the amendments DID change what the gate is reviewing.
