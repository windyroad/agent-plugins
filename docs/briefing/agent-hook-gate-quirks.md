# Agent + Hook Gate Quirks

Cross-session learnings about gate-marker propagation, create-gate helper bugs, and architect-agent gate visibility within AFK iteration subprocesses. Sibling to `agent-interaction-patterns.md` (broader interaction-discipline surface). Split out 2026-05-03 per P145 MUST_SPLIT contract for Tier 3 budget compliance.

## What Will Surprise You

- **Bypassing `/wr-itil:manage-problem` by writing directly to `docs/problems/` ships duplicates** (P119 verifying). The skill's Step 2 keyword-grep over existing ticket filenames + bodies is load-bearing — it catches "this concept is already tracked" before a new file lands. Observed: three direct-Write tickets that were near-perfect duplicates of two long-Open tickets. Always invoke the skill even when the observation feels novel — the grep is fast, the duplicate-bypass cleanup cost is high. <!-- signal-score: +1 | last-classified: 2026-05-25 | first-written: 2026-04-25 -->
- **SKILL.md "default for this skill" routing can be wrong-by-design when persona ≠ runtime mode** (P122) — `/wr-itil:work-problems` once deferred 12 design questions because it followed a "JTBD-006 = AFK is the default" prose rule while the user was demonstrably at the keyboard. Runtime detection (subprocess-vs-main-turn per ADR-032, AskUserQuestion-availability, prompt recency) should drive the branch, NOT the skill name. When a skill says "default for this skill is X", verify the default is grounded in runtime detection, not a named persona; follow the runtime mode when they disagree. <!-- signal-score: +1 | last-classified: 2026-05-25 | first-written: 2026-04-26 -->
- **The create-gate marker SID is now resolved via a runtime-SID instrumentation hook** (P142 Phase 4 / ADR-050, shipped @windyroad/itil@0.23.3) — a PreToolUse hook writes `${TMPDIR}/itil-runtime-sid-${USER}-${PID}.current` from the hook stdin JSON, and `get_current_session_id()` reads it first (announce-marker priority as cold fallback). This retired the earlier failure mode where the helper's ASCII-first glob over `/tmp/<system>-announced-*` markers returned a stale session's UUID, causing the create-gate hook to deny the Write; the P144 manual-touch recovery dance is no longer needed. Verified clean in post-restart sessions. <!-- signal-score: 0 | last-classified: 2026-05-25 | first-written: 2026-05-03 -->
- **Architect-agent invocation from inside an AFK iteration subprocess does NOT always satisfy the architect edit-gate for that subprocess's next Write** (may compose with P111, verifying). Observed: architect verdict APPROVED, next Write still blocked; re-invoking the agent cleared it. Workaround: when the next Write after an architect-agent call still hits the gate, re-invoke the agent (~1 turn per affected file). Distinct from TTL expiry. <!-- signal-score: 0 | last-classified: 2026-05-25 | first-written: 2026-04-26 -->
- **PostToolUse `architect-mark-reviewed.sh` parses verdict text by literal grep** — an `ISSUES FOUND` substring anywhere in agent output blocks the marker even when the verdict on the proposed change is PASS (e.g. a heading flagging issues in the *surrounding* ADR landscape while the proposed write passes). The PASS check runs first but loses to a later `ISSUES FOUND` elif. Workaround: when the substantive verdict is PASS-with-non-blocking-follow-ups, re-invoke (or SendMessage) asking for an `Architecture Review: PASS` heading with follow-up notes beneath. The heading IS the signal. <!-- signal-score: 0 | last-classified: 2026-05-25 | first-written: 2026-05-11 -->
