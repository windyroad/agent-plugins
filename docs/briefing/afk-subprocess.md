# AFK and Subprocess Iteration

Learnings about `/wr-itil:work-problems` AFK loops, `claude -p` subprocess dispatch, and iteration-worker protocols.

> **Sibling briefs** (P145 Tier-3 split): `afk-subprocess-mechanics.md` (dispatch shape, JSON contract, cost metadata, SID quirks) + `afk-subprocess-recovery.md` (SIGTERM flush, iteration-worker discipline, mid-loop escalation). Read all three together.

## What You Need to Know

- **Parallel Claude sessions can create P-ticket numbering collisions.** `manage-problem` / `create-adr` compute `max(local, origin)+1` automatically; `work-problems` fetches origin at loop start. Outside the skills, `git fetch origin` and check the highest P-number before opening tickets.
- **ADR-020 auto-release is skipped inside AFK loops** (ADR-018 Step 6.5, avoids double-release). A fix scoped to non-AFK governance paths is NOT exercised by `/wr-itil:work-problems` iterations — when evaluating Fix Released tickets for closure, check which code path actually fired.
- **Verify Fix Released tickets from your own in-session evidence** before deferring to the user — CI passes, preflight runs, release behaviour, next-ID assignments, commit-gate scoring are all observable. Only leave `needs user verification` for genuinely user-side exercise (interactive invocations not run in session, plugin-cache-lag, other-project-only behaviours). At loop stop, proactively surface close-candidates whose paths were exercised. (verifying P048 + P053)
- **At AFK loop stop, surface outstanding design questions before `ALL_DONE`** — user-answerable blockers (naming, direction, pacing, scope) go in a batched `AskUserQuestion` (cap 4); "needs interactive input" is not terminal. (verifying P053)
- **Every AFK halt path that emits a final summary routes prior-iter accumulated user-answerable skips through Step 2.5b first** (Step 0 continuity/fetch-fail, Step 6.5 CI-fail/above-appetite, Step 6.75 dirty-unknown). Step 2.5b surfaces *prior-iter skips only* — it does NOT ask how to remediate the halt cause itself (those stay halt-with-bug-signal). (verifying P122 + P126)
- **An iter returning `is_error: true` (API stream idle timeout) with coherent staged work is SALVAGEABLE — don't just halt.** Contractual Step 5 branch (ADR-032 § P261 amendment): IF `is_error: true` AND staged files exist AND iter-authored bats pass → the orchestrator commits the staged work from its OWN main turn with iter-attribution (commit gate fires fresh on its own SESSION_ID, per ADR-009); ELSE halt. The salvage commit IS the iteration's one commit. Distinct from P121 SIGTERM (subprocess had committed), P147 stuck-before-emit (0-byte JSON, metadata lost), P146 (bash-polling deadlock). The single bounded exception to "orchestrator does NOT commit from its main turn". (verifying P261) <!-- signal-score: +2 | last-classified: 2026-05-26 | first-written: 2026-05-26 -->
- **`API Error: The socket connection was closed unexpectedly` is a THIRD subprocess failure class — `is_error: true` with no staged work, P261 salvage does not apply.** Distinct from P261 (stream-idle-timeout, staged work survives), P147 (SIGTERM stuck-before-emit, 0-byte JSON), and P121 (SIGTERM idle-timeout, subprocess had committed). The socket-closed exit produces a coherent `.afk-run-state/iter-*.json` (metadata + cumulative cost preserved per Step 5 Authority Hierarchy) but no staged or committed files. Observed 2x consecutively in 2026-06-10 work-problems session at 9 + 23 turns — both well under the SIGTERM 3600s threshold, so the idle-timeout mechanism did NOT fire (subprocess exited on its own). Orchestrator main-turn is unaffected; the failure path is specific to `claude -p` subprocess transport. No documented retry policy yet — see P358 for investigation. - **Headless `claude -p` activates ONLY user-scoped `enabledPlugins` — project-scoped plugins stay dark unless passed via `--plugin-dir`.** Project-plugin activation is trust-gated and headless skips the trust prompt, so an AFK iter subprocess in a project that enables the windyroad plugins at PROJECT scope gets no architect/jtbd/risk/voice-tone agents or gate hooks — it commits ungated and can't run retro-on-exit (`Task(wr-architect:agent)` → "agent type not found"). `--setting-sources user,project` does NOT fix it (it doesn't lift the trust gate); only `--plugin-dir <plugin-root>` does (verified upstream 2026-06-21). work-problems Step 5 now derives the dirs portably via `wr-itil-resolve-governance-plugin-dirs` (cache-parent highest-semver walk, ADR-080 — NOT `$PATH` order, which is frozen-stale mid-session) and splices `--plugin-dir` pairs into the dispatch. ## What Will Surprise You

- **AFK iter subprocesses can over-claim completion in their `ITERATION_SUMMARY`** — iter 1 of session 8 (P327 / ADR-077 Slice 3) commit message said "all ADR-077 Confirmation items (a)-(j) green at source" yet on-disk the (a)-(j) boxes were all `[ ]` unchecked AND the compendium was NOT regenerated + staged despite Slice 3 implementing exactly that integration in `/wr-architect:review-decisions`. The CI drift gate this iter just shipped failed on the un-regenerated compendium — silent self-contradiction. Cross-check iter claims against on-disk state at Step 6.75 before trusting the summary; consider a `verify-iter-claims` step that greps the iter's stated artefacts against the actual repo.
