# Plugin Distribution — Cache and Install Mechanics

Sibling of `plugin-distribution.md`. Install / uninstall / refresh / cache / worktree / TUI-vs-agent quirks. The parent file retains stable distribution facts.

## What You Need to Know

- **Plugin hooks run from the marketplace cache, not from source.** After fixing hook code: push, `claude plugin marketplace update windyroad`, reinstall (`--scope project`), restart. Install at project scope to avoid breaking other active projects (ADR-004).
- **Plugin renames need 4 coordinated steps**: rename + push + release; `claude plugin marketplace update windyroad`; edit `.claude/settings.json` directly to swap the enabled key (`claude plugin uninstall` refuses project-scope); restart. Then `npm deprecate "<old-pkg>@>=0.0.0" "<msg>"` (bare `*` returns 404).
- **plugin.json top-level `hooks:` / `skills:` / `agents:` keys are RESERVED by the manifest validator** — arbitrary content there makes `claude plugin install` reject the plugin. Nest extensions UNDER an allowed key (e.g. top-level `maturity:` per ADR-063 Amendment). Validator runs at INSTALL time, not runtime — pre-existing installs keep working; new installs fail. (P263 tracks a CI gate)
- **`/install-updates` failure cascade can GUT `.claude/settings.json`** — when uninstall succeeds but install fails for N plugins, `enabledPlugins` is reduced to only fallback (non-windyroad) plugins. Recovery: `git checkout HEAD -- .claude/settings.json`. (P259)
- **`claude plugin install` can serve a STALE version-pinned cache dir even after `uninstall + install`.** Releasing `@windyroad/retrospective@0.21.0` then `/install-updates` left `~/.claude/plugins/cache/windyroad/wr-retrospective/0.21.0/` carrying the PRIOR version's hook content -- the uninstall removed the settings registration but NOT the cache dir, so install reused stale files (a P106/P259 facet). The published npm tarball AND the marketplace clone were both correct; only the cache dir was stale. Fix: `rm -rf ~/.claude/plugins/cache/windyroad/<key>/<version>` then reinstall to force a fresh copy. Verify post-install by grepping the cache dir's actual file content against the npm tarball -- NOT the version label (the label lies). ## What Will Surprise You

- **The marketplace resolves from the remote GitHub repo, not the local working tree.** You cannot install a new plugin until changes are pushed and `claude plugin marketplace update windyroad` pulls the latest. (Marketplace skills now show in autocomplete — #35641 fixed; the `skills` npm package workaround is retired per ADR-003.)
- **`claude plugin install <plugin>` is a SILENT NO-OP when the key is already installed at any version** — reports `✔ ... already installed`, exits 0 without fetching latest; cache stays on the old version. `claude plugin update` REJECTS project-scoped installs (user-scope only). The working refresh is `uninstall + install`, forcing a fresh marketplace download. (closed P106)
- **The plugin install CACHE (`~/.claude/plugins/cache/windyroad/<key>/<version>/`) is GLOBAL/shared across all projects — there is no per-project cache.** `.claude/settings.json` only *enables* a plugin by marketplace ref (`wr-X@windyroad`) with NO version pin; the active version is whatever sits in the shared cache. So a `uninstall + install` refresh run from ONE project advances the cached version for EVERY project that enables that plugin. This is why `/install-updates` was simplified 2026-05-25 to a single current-project refresh (ADR-030/059 amend): the sibling-project install loop + cross-project consent gate were unnecessary. Corollary: `claude plugin uninstall/install` REORDERS `enabledPlugins` (moves the re-added key to the end) — pure churn; revert with `git checkout .claude/settings.json` after a refresh (P280). <!-- signal-score: 0 | last-classified: 2026-05-25 | first-written: 2026-05-25 -->
- **Active sessions keep running the OLD cached plugin code until restart.** A cache refresh (even a just-shipped gate-hook fix) does not take effect in the current session — the hooks were loaded at session start. Practical consequence: when a subprocess authors changesets for a fix to the very gate it's hitting, it must apply the existing-gate workaround (the fix isn't live in its own session). The fix goes active on the NEXT session start. <!-- signal-score: 0 | last-classified: 2026-05-25 | first-written: 2026-05-25 -->
- **`claude plugin list` shows installs across ALL projects** (N entries = N project-scope installs per ADR-004, not duplicates; check `installed_plugins.json` for each `projectPath`). Two entries at *different versions* mean a stale install outlived its project path — most commonly an abandoned git worktree under `.claude/worktrees/<name>`, the smoking gun for silent-autocomplete-shadowing.
- **`installed_plugins.json` accumulates worktree-scope installs indefinitely — `git worktree remove` does NOT cascade-cleanup the registry.** Stale rows stay enabled until `claude plugin uninstall --scope project` is run *from inside the worktree path* (before removing it) or hand-edited out: `(cd <worktree> && claude plugin uninstall <plugin>@<marketplace> --scope project)` then `git worktree remove`. (closed P113; P115 tracks install-updates worktree scanning)
- **TUI autocomplete and the agent-side skill enumerator disagree under multiple install entries at different versions.** TUI uses the first-matching entry (earlier version wins → newer skills invisible in `/`); the agent enumerator unions them (all `Skill(...)`-invokable). When a skill appears "missing": check `claude plugin list` for duplicate entries and `installed_plugins.json` for stale rows BEFORE checking SKILL.md frontmatter. Upstream anthropics/claude-code#52831.
- **Releasing a plugin in a session does NOT make it usable in that session** (corollary of the active-sessions entry above). Accurate framing is "I released it but did not install it," not "the cache is stale." (verifying P078)
