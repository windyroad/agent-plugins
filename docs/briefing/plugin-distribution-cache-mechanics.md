## What You Need to Know

- **Plugin hooks run from the marketplace cache**, not from source. After fixing hook code, you must: push, update marketplace (`claude plugin marketplace update windyroad`), reinstall (`claude plugin install ... --scope project`), and restart Claude Code.
- **Install at project scope** (`--scope project`) to avoid breaking other active projects. ADR-004 documents this.
- **Plugin renames need 4 coordinated steps**: (1) rename in repo + push + release, (2) `claude plugin marketplace update windyroad` to refresh the cache, (3) edit `.claude/settings.json` directly to swap the enabled key (`claude plugin uninstall` refuses project-scope installs), (4) restart Claude Code. Then `npm deprecate "<old-pkg>@>=0.0.0" "<msg>"` — bare `*` returns 404; needs an explicit version range.
- **plugin.json top-level `hooks:` / `skills:` / `agents:` keys are RESERVED by Claude Code's manifest validator** — only specific event-keyed schemas accepted. Adding arbitrary content (maturity records, custom metadata, anything else) at those keys causes `claude plugin install` to reject the plugin with `Validation errors: hooks: Invalid input, skills: Invalid input`. Workaround: nest extensions UNDER an allowed key (e.g. top-level `maturity:` for maturity-records per ADR-063 Amendment 2026-05-18). The validator runs at INSTALL time but not at runtime — pre-existing installs continue to work; new installs fail. Observed 2026-05-18 P0 — 11 plugins shipped broken via Phase 3 retroactive rollout (commit d33bb7d → @windyroad/itil@0.35.1 + 10 siblings); hotfix in commit 3cfa6fc (ADR-063 Amendment + schema_version 2.0 bump) restored validity. CI gate to prevent recurrence tracked at P263.
- **`/install-updates` failure cascade can GUT `.claude/settings.json`** — when uninstall succeeds but install fails for N plugins (e.g. all plugins via a broken-manifest release), the settings's `enabledPlugins` map is reduced to only fallback plugins (non-windyroad). Recovery: `git checkout HEAD -- .claude/settings.json`. Observed 2026-05-18 — 11 wr-* plugins stripped from settings during the P258 broken-manifest install attempt. Defensive snapshot-and-restore tracked at P259.

## What Will Surprise You

- **anthropics/claude-code#35641 is fixed** — marketplace skills show in autocomplete. The `skills` npm package workaround is no longer needed (ADR-003).
- **The marketplace resolves from the remote GitHub repo**, not the local working tree. You cannot install a new plugin until changes are pushed and `claude plugin marketplace update windyroad` pulls the latest.
- **`claude plugin install <plugin>` is a SILENT NO-OP when the plugin key is already installed at any version.** The CLI reports `✔ Plugin ... is already installed (scope: project)` and exits 0 without fetching the latest version from the marketplace. The cache dir stays on the old version. Observed 2026-04-22 across 24 project/plugin combos (6 projects × 4 updated plugins) — every `install` call no-opped; `@windyroad/retrospective` cache stayed on 0.6.0 despite npm latest at 0.8.0. `/install-updates` Step 7 currently uses this command and therefore silently does nothing after a release (P106).
- **`claude plugin update <plugin>` exists but REJECTS project-scoped installs** with `"Plugin ... is not installed at scope user"`. `update` is user-scope only. ADR-004 mandates project-scope for all windyroad plugins, so `update` is not a workable refresh path.
- **The working refresh pattern for project-scoped plugins is `uninstall + install`.** `claude plugin uninstall <plugin>@windyroad --scope project` followed by `claude plugin install <plugin>@windyroad --scope project` forces a fresh marketplace download. Tested 2026-04-22 across all 24 combos: every one refreshed cleanly to the npm latest version. P106 tracks the `/install-updates` Step 7 fix.
- **The plugin install CACHE (`~/.claude/plugins/cache/windyroad/<key>/<version>/`) is GLOBAL/shared across all projects — there is no per-project cache.** `.claude/settings.json` only *enables* a plugin by marketplace ref (`wr-X@windyroad`) with NO version pin; the active version is whatever sits in the shared cache. So a `uninstall + install` refresh run from ONE project advances the cached version for EVERY project that enables that plugin (they all read the same cache on next session). This is why `/install-updates` was simplified 2026-05-25 to a single current-project refresh (ADR-030/059 amend): the sibling-project install loop + cross-project consent gate were unnecessary — there are no per-project cache writes to consent to, and the refresh benefits siblings automatically. Corollary: `claude plugin uninstall/install` REORDERS `.claude/settings.json` `enabledPlugins` (moves the re-added key to the end) — pure churn, no semantic change; revert with `git checkout .claude/settings.json` after a refresh (P280). <!-- signal-score: 0 | last-classified: 2026-05-25 | first-written: 2026-05-25 -->
- **Active sessions keep running the OLD cached plugin code until restart.** A cache refresh (even a just-shipped gate-hook fix) does not take effect in the current session — the hooks were loaded at session start from the then-current cache. Practical consequence: when a subprocess authors changesets for a fix to the very gate it's hitting, it must apply the existing-gate workaround (the fix isn't live in its own session). The fix goes active on the NEXT session start. <!-- signal-score: 0 | last-classified: 2026-05-25 | first-written: 2026-05-25 -->
- **`claude plugin list` shows installs across all projects, not just the current one.** If you see N entries for a plugin, they are N different project-scope installs (correct per ADR-004), not duplicates. Check `~/.claude/plugins/installed_plugins.json` for the `projectPath` of each entry. **Caveat**: if two entries show *different versions* of the same plugin, at least one is a stale install that outlived the project path it was registered against — most commonly an abandoned git worktree under `.claude/worktrees/<name>`. This is the smoking gun for the silent-autocomplete-shadowing class (P113 cause).
- **`installed_plugins.json` accumulates worktree-scope installs indefinitely — `git worktree remove` does NOT cascade-cleanup the registry.** Each git worktree registers as its own project scope for Claude Code's plugin registry. Worktree-scope installs persist in `~/.claude/plugins/installed_plugins.json` even after `git worktree remove --force` + `git branch -d`; they stay flagged enabled until an explicit `claude plugin uninstall --scope project` is run *from inside the worktree path* (before removing it) or the row is manually edited out of the JSON. Per-machine cleanup pattern: `(cd <worktree> && claude plugin uninstall <plugin>@<marketplace> --scope project)` then `git worktree remove`. Observed 2026-04-24 during P113: an abandoned `.claude/worktrees/upbeat-keller` (installed 2026-04-18 when `wr-itil` was at 0.1.0) had its 0.1.0 install still registered six days later, silently shadowing the outer project's 0.18.1 for TUI slash-command autocomplete while the agent-side enumerator happily saw the 0.18.1 skills. Systemic fix tracked in P115 (install-updates should scan worktrees).
- **Claude Code's TUI slash-command autocomplete and agent-side skill enumerator read different code paths when a plugin has multiple install entries at different versions — they can silently disagree.** Observed behaviour: TUI appears to use the first-matching install entry's skill list (earlier version wins → skills added in later versions are invisible in the `/` dropdown); agent-side enumerator takes the union across versions (all skills remain `Skill(...)`-invokable). The asymmetry produces a confusing failure mode where the agent insists a skill exists while the user cannot discover it via `/`. Reported upstream as anthropics/claude-code#52831 (2026-04-24). Diagnostic order when a skill appears "missing" in autocomplete: (1) check `claude plugin list` for duplicate entries at different versions; (2) grep `~/.claude/plugins/installed_plugins.json` for stale rows; (3) only THEN investigate plugin-side SKILL.md frontmatter — the plugin side is rarely the cause.
- **Releasing a plugin in a session does NOT make it usable in that session.** Slash commands and skills load their SKILL.md content from `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/`. The cached version only advances when `/install-updates` runs AND the session is restarted. If you release `@windyroad/retrospective@0.5.0` mid-session, invoking `/wr-retrospective:run-retro` will still execute the previously-installed version's SKILL.md (e.g. `0.3.0`). Symptom: retro runs the legacy flow even though you just shipped the new one. Fix: run `/install-updates` + restart before relying on any self-shipped behaviour. **Do not frame this as "the cache is stale" — that is passive-voice dodging. The accurate framing is "I released it but did not install it."** Observed 2026-04-21 this session. See P078 for the proactive-offer-ticket-on-user-correction gap that let this dodging reach the user.
