# Agent Interaction Patterns

Cross-session learnings about the assistant own interaction habits — framing-validation, re-stage traps, user-frustration signals, and the small rules that reduce wasted rounds.

## What Will Surprise You

- **Write tool fails with "File has not been read yet" after `git mv`.** Renaming a file mid-session invalidates the Read-tracking on the new path. Must `Read` the renamed file before `Write` or `Edit`.
- **`git mv` + `Edit` + `git add` requires re-stage after the Edit.** `git mv A B` places the rename in the index. If you then use the Edit tool on B and later run `git add <other-files>` before the final commit, that captures only the rename — the Edit to B stays unstaged and lands in the NEXT commit. Must `git add B` (or include B explicitly in the final add-list) after editing. Adjacent to the "Write fails after git mv" line above but a distinct staging-ordering trap. Observed 2026-04-19 iter 1: P054 verifying.md rename landed in 45e9c71, content edit (Status + Fix Released) only landed in iter 2 b2f1646 alongside P046.
- **Validate the problem ticket framing before asking solution-detail design questions.** When a ticket describes a specific solution (e.g. P014 described `/btw`-style stubs with backfill), the user may have a different shape in mind. First `AskUserQuestion` should confirm the framing, not dive into solution details. Observed 2026-04-20 (first incident): P014 four detail questions (stub format, backfill trigger, context escape, dispatcher) were framed around stubs; user reframed as "no stubs, auto-delegate on every invocation" in one Other-response. Observed 2026-04-20 (AFK stop-condition #2 retro): P014 three shape options (inline structured / out-of-band subagent / skill-wrapped) were framed around new patterns; user reframed again as "just use background agents" — collapsed scope from XL to L (WSJF 1.5 -> 3.0) and eliminated the ADR. Other-responses keep recovering from framing misses; cost is one round of questions but the option saves the downstream skill invocation. **If the ticket has already been bounced by a framing miss once, open the next AskUserQuestion with a framing check ("Is this ticket shape still right — pattern, skill, agent, background subagent, or something else?") before detail questions.**
- **Bypassing `/wr-itil:manage-problem` by writing directly to `docs/problems/` ships duplicates.** The skill's Step 2 keyword-grep over existing `.open.md` / `.known-error.md` / `.closed.md` filenames + bodies is load-bearing — it catches "this concept is already tracked" before a new file lands. Observed 2026-04-25: the assistant wrote three new `.open.md` files (P119/P120/P121) directly via Write to capture #52831 retrospective findings; all three were near-perfect duplicates of P038 (no voice-tone gate on external comms) and P064 (no risk gate on external comms), both Open since 2026-04-17 with ADR-028 already landed. Cleanup cost: 3 file deletes + 2 evidence-append edits to the surviving tickets. **Always invoke `/wr-itil:manage-problem` even when the observation feels novel** — the grep is fast, the duplicate-bypass cost is high. The retro-skill Step 4b Stage 1 contract makes this mechanical for codification candidates; same discipline applies to ad-hoc ticket creation outside the retro flow.
- **User-frustration signal: "why are you asking me and not just doing it?"** Observed 2026-04-21 on the install-updates consent gate (mandated by ADR-030) when the answer was obviously "all 6 siblings". The pattern generalises: when a consent gate or a "proceed vs discard?" decision is obvious from direction-pinning, policy (risk within appetite), or homogeneous context, ACT first and report — do not defer to the user. This is the same shape as the halt-recovery frustration earlier the same session. Memory: `feedback_act_on_obvious_decisions.md`. Candidate follow-up: relax ADR-030 consent gate when sibling set is homogeneous + user already invoked `/install-updates` in wrap-up context.
- **Problem-ticket Fix Strategy can describe infrastructure that does not exist.** Observed 2026-04-26 P121 AFK iter 2: the ticket's Fix Strategy and Investigation Tasks both asserted "the orchestrator's existing poll loop already iterates in 60s ticks" and proposed extending it. The architect agent's pre-edit review verified the SKILL.md and confirmed there was NO poll loop — the actual Step 5 dispatch was a plain synchronous foreground `claude -p` call. The fix shape then changed from a prose-extension to a structural change (foreground → backgrounded + explicit poll loop) plus an ADR-032 amendment block. **Always have the architect verify the ticket's premise about existing infrastructure before drafting the fix** — ticket authors (often run-retro's Step 2b at session-end) describe what they assume exists rather than what the SKILL.md actually contains; the assumption can survive ticket creation, WSJF rating, and AFK orchestrator selection without challenge. Cost of skipping the check: a wasted iteration that ships the wrong shape; cost of the check itself: one architect delegation. Pattern overlap with the framing-validation rule above — both belong to "validate before implementing" but on different surfaces (user-framing-validation vs architect-premise-validation).
- **SKILL.md "default for this skill" routing rules can be wrong-by-design when persona ≠ runtime mode.** Observed 2026-04-26 P122: `/wr-itil:work-problems` Step 2.5 routing prose said *"JTBD-006's persona constraint makes the non-interactive path the default for this skill — AskUserQuestion is the exception, not the rule"*. The orchestrator (main turn, with `AskUserQuestion` available, user demonstrably at the keyboard) followed the default and emitted the deferred Outstanding Design Questions table for 12 questions across 5 skipped tickets. User correction within seconds: *"this was a really good example of where there are many outstanding design questions, but nothing was asked"*. The bug: persona (`JTBD-006 = AFK`) was conflated with runtime mode. The runtime detection (subprocess-vs-main-turn per ADR-032; AskUserQuestion-availability; recency-of-prompt) should drive the branch — NOT the skill name. **When a skill prose says "default for this skill is X", verify the default-selection rule is grounded in runtime detection, not in named persona.** When the actual runtime mode disagrees with the named persona, follow the runtime mode. P122 captures the SKILL.md-side fix; the rule generalises to any AFK-named skill that the user invokes interactively.
- **`docs/problems/` create-gate hook (P119) requires the Step 2 marker named with the actual session UUID — `${CLAUDE_SESSION_ID:-default}` evaluates to `default` and the hook denies.** Observed 2026-04-26 P122 retro: the SKILL.md fallback chain produces the wrong marker name; the hook reads `session_id` from its stdin JSON payload (the actual UUID) and the names mismatch. **Discovery pattern: scrape an existing `/tmp/architect-plan-reviewed-<UUID>` (or any other reliably-set gate marker) to extract the UUID, then `: > "/tmp/manage-problem-grep-<UUID>"`.** The architect marker is set by every architect-gate-passing edit so it's typically the most-recent + reliably-present marker for any session that's done some work. P124 captures the SKILL.md improvement (helper script + cited discovery contract).
