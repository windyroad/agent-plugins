# Changeset Holding and Graduation

Cross-session learnings about `docs/changesets-holding/` (ADR-042 Rule 7), held-cohort graduation, and what holding does and does not control. Split out of `releases-and-ci.md` 2026-06-11 per Tier 3 budget rotation.

## What You Need to Know

- **The changeset-discipline gate (Check 2b) is now CHANGE-scoped, not plugin-scoped (P387, itil patch pending release 2026-06-27).** A `packages/<plugin>/` source commit no longer rides an UNRELATED in-scope changeset's coattails. Check 2b denies only on positive unrelated-sibling evidence: the commit message cites a work-item ID (`P<NNN>`/`RFC-<NNN>`/`STORY-<NNN>`) AND every in-scope changeset covering that plugin cites work-item ID(s) AND none overlap. Allow-on-ambiguity otherwise — a ticket-less commit, a prose-only changeset (no ID in filename/body), or an ID overlap all PASS. Practical impact in multi-ticket sessions: when you commit a P-X fix while a sibling P-Y changeset for the same plugin is already in scope, you must author P-X's OWN changeset (Check 2a) OR ensure a covering changeset cites P-X — otherwise the gate denies. The ADR-014 batch-grain still works because same-slice multi-commits share the slice's ticket. The hook extracts the commit ID from the `git commit` command it already parses (filename + body on the changeset side); ADR refs are deliberately NOT matched (cross-cutting, not change-identity). `changeset-detect.sh` is NOT a synced hook — single copy in `packages/itil/hooks/lib/`. See P141 Phase 3. <!-- signal-score: 0 | last-classified: 2026-06-27 | first-written: 2026-06-27 -->
- **Atomic-cohort full graduation scales empirically to 13 changesets / 5 packages with no false-positives** (ADR-060 finding 12). Graduate a held cluster atomically (`git mv docs/changesets-holding/*.md .changeset/`) on user-comfort signal, not elapsed time. (verifying P162)
- **`.changeset/` empty + `docs/changesets-holding/` non-empty can self-reinforce a release-pressure stall**: ADR-042 auto-moves every changeset to holding, so the ADR-018/020 drain (needs `.changeset/` non-empty) never fires — commits accumulate unpushed AND nothing publishes. Only manual graduation recovers it. Monitor held-cluster size + last-push age proactively. (verifying P162)
- **Dogfood criterion is positive-evidence-of-working, NOT elapsed time.** Graduate a held entry when its surface fired correctly with no false-positive; KEEP HELD when it has negative evidence (e.g. hook misfired), regardless of days. Per-entry evidence beats sibling-cohort piggybacking. (verifying P246) <!-- signal-score: 1 | last-classified: 2026-06-11 | first-written: 2026-06-11 -->

## What Will Surprise You

- **Moving a changeset to `docs/changesets-holding/` does NOT keep the code from shipping — it only withholds version attribution + CHANGELOG entry.** npm publishes main's package directory contents; any sibling-changeset-driven release ships ALL committed code, held or not. Witnessed 2026-06-11 (P220 AFK iter): the P220 Step 0d fix's changeset was held 2026-06-08, yet `npm pack @windyroad/itil@0.49.3` contains every fix file (SKILL prose + lib + script + bin shim) because 0.48.0/0.49.x released sibling changesets after the fix commit landed on main. The ADR-042 hold is changelog/attribution governance, NOT a shipment control — an above-appetite change held "out of the release queue" still reaches adopters on the next sibling release. Ticket captured 2026-06-11 for the framework gap. <!-- signal-score: 0 | last-classified: 2026-06-11 | first-written: 2026-06-11 -->
- **"Cohort graduated" claims can be PARTIAL — verify each changeset file's location, never trust the cohort label.** The 2026-06-11 0.49.4 release drained 4 of the 6 work-problems-cohort held changesets (P206/P211/P212/P228) but left `p220-step-0d-outbound-responses-preflight.md` + `p214-step-5-is-error-transient-halt.md` still in `docs/changesets-holding/` despite the same reinstate criterion (0.47.15 eval) covering all 6. Before asserting "attribution shipped" for ticket N, check `ls docs/changesets-holding/ .changeset/` for N's specific file. K→V transitions stay valid either way (code ships regardless, see entry above); only the changelog-attribution claim needs the per-file check. (witnessed AFK iter 2, 2026-06-11) <!-- signal-score: 0 | last-classified: 2026-06-11 | first-written: 2026-06-11 -->

- **The R009 promptfoo-eval floor is the recurring reason SKILL/agent-prose changesets get held — and eval cohorts discharge MANY at once.** A SKILL.md/agent.md prose change with no paired promptfoo Tier-A/B eval scores at the R009 prose floor (≈6-8/25, above the 5/Low appetite) → move-to-holding per ADR-042 Rule 2. These accumulate PER SKILL surface: by 2026-06-28 the `work-problems` surface had **9 distinct held SKILL-prose fixes** (P206/P211/P212/P214/P220/P228/P352/P358/P390) and `capture-problem` had P199/P350/P383 — all on the SAME floor. Every new work-problems SKILL-prose fix adds another hold until the one `work-problems` eval is run GREEN, so the cohort only ever deepens under AFK iters (each iter authors its paired case + holds the changeset, never runs the suite). Leverage: ONE eval-authoring pass per surface (extend `packages/itil/skills/<skill>/eval/promptfooconfig.yaml`, run `npx promptfoo eval` GREEN) discharges the whole cohort's floor; the held changesets then graduate together. Caveats: (a) `eval/` is NOT in the changeset-discipline allow-list, so an eval-config edit triggers its own changeset (P387-adjacent); (b) HELD-cohort graduation is interactive-only (Rule 4 / P308) even once the eval is green — AFK can only author+green the eval and QUEUE the graduation; (c) one red case blocks the whole cohort's "eval GREEN" criterion (capture-problem ran 7/8 on a pre-existing mis-calibrated case → P388 → cohort stays held). Eval runs are expensive (~$10-20: each case spawns `claude -p` + a Tier-B grader call). See `promptfoo-eval-authoring.md`, P388, ADR-042, ADR-061. (2026-06-28.) <!-- signal-score: 2 | last-classified: 2026-06-28 | first-written: 2026-06-27 -->

- **Moving the changeset to holding does NOT unblock the COMMIT — the commit risk gate scores the in-source SKILL prose, not just the release queue.** A SKILL-prose fix held per ADR-042 Rule 2 still trips the `git commit` risk gate at the R009 in-source floor (~6-8/25) because the gate scores the staged SKILL.md prose entering the source tree, independent of whether a `.changeset/` entry is queued. Holding zeroes the RELEASE layer only; the COMMIT/PUSH layers carry the in-source prose-defect residual. To land the commit you must EITHER (a) author the paired promptfoo case AND run it GREEN — that discharges the in-source floor (modulator +1→−1) so `wr-risk-scorer:pipeline` re-scores within appetite and authors the reducing-bypass marker — OR (b) qualify for a reducing-bypass another way (e.g. the commit CLOSES a Known Error → `.closed.md` rename). An Open→Known-Error transition alone does NOT qualify under the tightened post-P329 reducing-bypass criteria (a fresh scorer agent will refuse the bypass citing "not → Closed"). Practical AFK consequence: the "fix lands, changeset holds" pattern is only half-true — the fix lands ONLY after you run the eval green (or close a KE); a held changeset by itself leaves the commit gate blocked. Witnessed P371 iter 2026-06-28: held changeset scored release=0 but commit=6 until the manage-problem P371 eval case ran 3× GREEN → re-score commit=4 → reducing-bypass authored → commit landed. See `hooks-and-gates.md` (commit risk gate), ADR-042, ADR-061. (2026-06-28.) <!-- signal-score: 2 | last-classified: 2026-06-28 | first-written: 2026-06-28 -->
