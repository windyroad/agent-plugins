STORY-MAP-002: Decompose a Fix Into Coordinated Changes

A Jeff Patton user story map for JTBD-008 — the developer's full end-to-end journey, from "something hurts" to "the adopter gets a real, coordinated fix." Backbone × ribs × slices per Patton, User Story Mapping (O'Reilly, 2014).

When my fix to a problem decomposes into multiple coordinated changes, I want the work scoped, time-boxed, and traced to its driving problem above the commit level — so each sub-workstream competes for WSJF attention as a first-class entity, not buried in a ticket body.

Persona: developer (solo developer using AI agents to investigate, fix, and ship end-to-end) · secondary: tech-lead.

End outcome (where the backbone lands): the adopter who hit the problem receives a coordinated, RFC-backed fix in a published @windyroad/* release — with a durable problem → RFC → story trace anyone can re-derive later.

Actor note. The backbone is one persona's user activities (the developer), left→right as a narrative with a beginning, middle, and end. The AFK orchestrator (JTBD-006) walks the same spine and lawfully halts at the activities where a human decides (scoping, uncovered-decision escalation, verification). Cards are user-observable capabilities; the implementing STORY / RFC / ADR is a small reference on the card, never its headline.

Side-rail — migration & debt (off the happy-path journey)

Who: maintainer. Why: the repudiated "auto-create a skeleton RFC at fix-time" mechanism left hollow traces behind. Pain if ignored: a future maintainer or agent trusts a structurally-present but empty RFC trace (P375 cadence-rot). This work does not advance any backbone activity — it cleans up after a superseded approach.

STORY-017 Backfill-or-supersede the skeleton RFCs the auto-create mechanism left behind ADR-031 migration Adopter docs/problems/ flat → per-state-subdir auto-migration (composes with JTBD-101)

Where RFC-005 sits. RFC-005 (RFC-first enforcement) and its STORY-012..017 are one slice cluster within Activities 3–4 of this journey — the part that guarantees the fix actually goes through a real, pre-existing RFC before any code is written. They are not the whole job; the backbone spans capture → root cause → propose → scope → implement → release & verify. RFC-003 / STORY-001..007 built the framework plumbing (capture-story, manage-story, work-the-problem traversal) that Activities 4–5 reuse.
