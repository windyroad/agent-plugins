STORY-MAP-002: RFC-first enforcement at fix-time

How to read this map (the two layers). The backbone is the maintainer's journey fixing a problem once RFC-first is enforced — user activities in narrative flow, ending in adopter value (per Patton: the backbone is user activities, not developer tasks). Each slice states the user-observable capability that step gains, and links the STORY-NNN ticket that implements it — the build work lives in the ticket, not on the spine. Releases are horizontal slices, each with a target outcome; Release 1 is the walking skeleton — a thin, operable, proven thread across every activity.

Primary user: the plugin maintainer. Second actor: the AFK orchestrator (/wr-itil:work-problems), which walks the same spine unattended and lawfully halts where a human would decide. Conventions: grey = existing capability reused (no new build); dashed = a path variation (no separate ticket).

Backbone — the maintainer's journey fixing a problem under RFC-first (left → right, ending in adopter value)

A1 · Propose a fix & hit the gate
A2 · Author the RFC (or escalate a new decision)
A3 · Implement the RFC's stories
A4 · Land the fix → adopter gets a real-plan-backed fix

Release 1 — walking skeleton (thin, operable, proven across every activity)

Target outcome: a maintainer can fix a covered problem RFC-first, end-to-end — and it's proven. Success metric: the predicate / refuse-route bats are green AND the author→implement→land thread is dogfooded green on a real covered problem.

A1 maintainer + AFK — The gate refuses my ad-hoc fix and routes me to author an RFC first. Implemented by STORY-012 (predicate signal) + a thin slice of STORY-013 (the gate that acts on the signal & routes) — together, operable. A2 maintainer — I author one minimal RFC as a story map for my covered fix (happy path: existing ADRs cover it). STORY-015 (thin: capture-rfc covered path). A3 maintainer — I implement the RFC's first not-done story. (existing) work-the-problem traversal, RFC-003 / STORY-005 — reused, no new build.
A4 maintainer → adopter — The fix lands (Known Error→Verifying→Closed) and the adopter gets a correctly-fixed plugin backed by a real, traced plan. (existing) ADR-022 lifecycle — this is the value RFC-first exists to deliver.

Quality (cross-cuts A1–A4) — Release 1 proof

Proven: behavioural bats cover the predicate + refuse/route layer of the R1 thread; the author-first routing & authoring (SKILL-orchestration) is proven by the STORY-015 forward-dogfood — an honest harness-gap, not a regression test (disclosed in STORY-016). STORY-016 (core).

Release 2 — deepen the same activities (not new ones)

Target outcome: the gate is complete — the AFK actor, the uncovered-decision path, and richer authoring all work. Success metric: bats green for uncovered→blocks-for-ADR, AFK lawful-halt, and refusal at every effort level.

A1 maintainer — The full interactive gate: a pre-existing RFC lets me proceed to implement; a missing one routes me to author-first. STORY-013 (full). A1 AFK orchestrator — Unattended, the gate authors-first for covered fixes and lawfully HALTS the loop on an uncovered decision (the agent's divergence from the human). STORY-014.
A2 maintainer / AFK — variation (happy): the fix approach is covered by existing ADRs → cite them & proceed, no new ADR (P132).
A2 maintainer — variation (unhappy, richest path): the approach is uncovered → STOP, author a new ADR, ratify it before implementing (ADR-070/073). The AFK actor halts here.
A2 maintainer — Richer authoring via manage-rfc; the --fix-time byproduct path is retired entirely. STORY-015 (deepen).
A4 maintainer — variation (unhappy): verification fails → the fix bounces back to A2/A3 (re-author or re-implement the RFC's stories) before it can close.
quality — Deepened proof: bats for uncovered→blocks-for-ADR, covered→proceeds, refusal at every effort level (S/M/L/XL). STORY-016 (deepen).

Migration side-rail — off the maintainer's journey (one-off cleanup)

Whose pain: a future maintainer or AFK agent who trusts a problem's trace that is structurally present but hollow (an empty skeleton RFC) — the P375 cadence-rot. Outcome: every trace points at a real plan.

Backfill-or-supersede the skeleton RFCs the repudiated auto-create mechanism left behind (RFC-026/028/029/030/032/033/034). A debt sweep that runs independently of the journey. STORY-017.

Releases are depth, not new activities. Release 1 is the thinnest operable thread across all four activities (refuse+route, author one covered RFC, implement, land — proven by its own bats), so a maintainer can already fix a covered problem RFC-first end-to-end. Release 2 deepens the same activities (full + AFK gate, the uncovered→ADR path, richer authoring, the verify-fail loop, fuller proof). The migration side-rail is off the journey. Structure per Patton, User Story Mapping (O'Reilly, 2014): backbone = user activities (unprioritised); ribs/releases = prioritised slices; walking skeleton = thin-but-operable across the whole backbone.
