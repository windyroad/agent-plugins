STORY-MAP-002: RFC-first enforcement at fix-time

This maps the maintainer's journey fixing a problem once RFC-first is enforced — and slices the build (RFC-005) so the thinnest first release delivers a usable, proven RFC-first loop end-to-end. Backbone (user activities) × release ribs × slices per Patton, User Story Mapping (O'Reilly, 2014).

Primary user: the plugin maintainer fixing a problem. Second actor: the AFK orchestrator agent (/wr-itil:work-problems), which walks the same journey unattended and lawfully halts where a human would decide. Each slice names its actor. Cards link their INVEST story; dashed cards are path variations (no separate story); grey cards are existing capabilities RFC-first reuses (no new build).

Backbone — what the maintainer does to fix a problem (left → right)

A1 · Propose a fix & hit the gate
A2 · Author the RFC (or escalate a new decision)
A3 · Implement the RFC's stories
A4 · Land the fix

Release 1 — walking skeleton (thin, proven, end-to-end: hit the gate → routed to author → author one RFC → implement a story → land)

A1 maintainer + AFK — STORY-012 (core): the gate detects no RFC traces the problem and refuses / routes to authoring. A2 maintainer — STORY-015 (core): author one minimal RFC as a story map from the problem's RCA (happy path: fix is covered by existing ADRs). A3 maintainer — (existing) implement the RFC's first not-done story via the work-the-problem traversal (RFC-003 / STORY-005). RFC-first reuses it.
A4 maintainer — (existing lifecycle, ADR-022) the fix lands: Known Error → Verifying → Closed. The payoff RFC-first delivers: the problem is now fixed by a real, pre-existing RFC (a genuine plan with traced stories) instead of a hollow ad-hoc ## Fix Strategy checklist. No new build — this verified, closed-with-a-real-plan outcome is the value the whole gate exists to protect.

Quality rib (cuts across A1–A4) — Release 1

STORY-016 (core): behavioural bats prove the R1 thread — an RFC-less Known Error is refused / routed end-to-end. (R1 ships a proven gate, not an unproven one.)

Release 2 — deepen the gate; the variations & unhappy paths (breadth under A1 / A2)

A1 maintainer — STORY-013: full interactive propose-fix gate (pre-existing RFC → implement; none → author-first). A1 AFK orchestrator — STORY-014: author-first for covered fixes; lawfully HALT the loop on an uncovered decision (the agent's divergence from the human).
A2 maintainer / AFK — variation (happy): fix approach is covered by existing ADRs → cite them & proceed, no new ADR (P132). (Acceptance in STORY-013/014.)
A2 maintainer — variation (unhappy / richest path): fix approach is uncovered by existing ADRs → STOP, author a new ADR, ratify it before implementation (ADR-070/073). The AFK actor halts here.
A2 maintainer — STORY-015 (deepen): richer authoring via manage-rfc; retire the --fix-time byproduct path entirely. quality — STORY-016 (deepen): bats for the unhappy paths — uncovered → blocks for ADR; covered → proceeds; refused at every effort level (S/M/L/XL).

Migration side-rail — NOT on the maintainer's journey (one-off cleanup, runs independently)

STORY-017: backfill-or-supersede the skeleton RFCs the repudiated auto-create mechanism left behind (RFC-026/028/029/030/032/033/034). A debt sweep, not a step any maintainer walks.

How to read the releases (depth, not new activities): Release 1 is the thinnest thread that crosses every backbone activity and is proven by its bats — a maintainer can already fix a covered problem RFC-first end-to-end. Release 2 deepens the same activities (full gates, the AFK halt, the uncovered→ADR unhappy path, richer authoring). The migration side-rail is off the journey entirely.

Driving problems: P251 (trace invariant not enforced), P314 (gate-design rework), P371 (existing-vehicle branch), P399 (skeleton-RFC repudiation). Decisions: ADR-071/072/073 (RFC-first), ADR-060 (RFC = story map), ADR-070 (uncovered option → ADR), ADR-022 (lifecycle). Sibling map: STORY-MAP-001 (RFC framework bootstrap).
