# Changesets Holding Area

> **Blessed by ADR-042** (2026-04-23). This is the authoritative mechanism for the `move-to-holding` remediation class under ADR-042's open vocabulary. See `docs/decisions/042-auto-apply-scorer-remediations-open-vocabulary.proposed.md` Rules 2 + 7.

Changesets that are authored against landed commits but are **not yet ready to ship** because they belong to a multi-slice fix whose other slices have not yet landed, OR that have been auto-moved here by the orchestrator under ADR-042 Rule 2 to bring release risk within appetite. Holding them here (outside `.changeset/`) keeps the intent intact without breaking the `changesets/action@v1` Release workflow, which does not tolerate subdirectories under `.changeset/` (`ENOENT` on `.changeset/pending/changes.md` observed 2026-04-22 — the original relocation target).

## When to hold a changeset

A changeset is a candidate for holding in either of two cases:

1. **Multi-slice WIP (user-authored)**: a `minor`/`major` bump that migrates file layout or introduces behaviour whose paired consumer-side hook lives in a slice still pending architect decisions — the P104 "painted into a corner" hazard.
2. **Auto-apply under ADR-042 (orchestrator-authored)**: residual push/release risk is above appetite (≥ 5/25) and the scorer suggests `move-to-holding` as a remediation. The orchestrator decides, performs the move, re-scores, and proceeds per ADR-042 Rule 2.

## Process

1. Author the changeset against the slice's commits normally in `.changeset/`.
2. When the slice-1-without-slice-N hazard is recognised (user path) OR the orchestrator's auto-apply fires (ADR-042 path), `git mv .changeset/<name>.md docs/changesets-holding/<name>.md`.
3. Reference the holding state:
   - User path: in the parent ticket's `## Fix Strategy` or `## Design Update` section so the reinstate trigger is captured.
   - Auto-apply path: the orchestrator's iteration/skill report logs the move per ADR-042 Rule 6, and this README's "Currently held" section is appended with the parent ticket reference.
4. When the blocking slices land (user path) or the user manually decides to reinstate (auto-apply path), `git mv docs/changesets-holding/<name>.md .changeset/<name>.md` and push. The next Release workflow run picks it up. Move the entry from "Currently held" to "Recently reinstated" in this README with the reinstate date + reason.
5. **Graduation (orchestrator path, per ADR-061; refined by P246; Rule 4 judgement locus clarified by P308)**: when the orchestrator is in within-appetite drain mode per ADR-018 / ADR-020 (Rule 8 amended drain condition — drain wakes when `.changeset/` is non-empty OR `docs/changesets-holding/` contains entries), the orchestrator invokes the risk-scorer's `wr-risk-scorer-evaluate-graduation` deterministic Rule 1a join + Rule 2 VP carve-out + Rule 3b cohort-grouping evaluator. The evaluator emits one `GRADUATION_CANDIDATE: changeset=<basename> | ticket=<P-id> | priority=<N> | class=<3a|3b> | [cohort=<id> |] status=<resolved|vp-blocked|halt-no-resolution>` line per held entry. The orchestrator branches per `status`: `resolved` → **route to Rule 4 evidence-floor judgement** (P308 — necessary-but-not-sufficient; see Rule 4 judgement-locus paragraph below); `vp-blocked` → skip (Rule 2); `halt-no-resolution` → halt-and-prompt per Rule 1a terminal (work-problems SKILL.md Step 6.5 cohort-graduation halt-no-resolution halt point). Cohort propagation (Rule 3b) graduates the entire cohort atomically when ALL same-cohort members reach Rule 4-ratified graduate; any Defer/Reject keeps the cohort held. After Rule 4-ratified graduate, the orchestrator proceeds to push/release per ADR-018/020 drain. The README audit trail (Currently held → Recently reinstated) is updated per ADR-061 Rule 6 with the cited evidence artefact, the resolved problem-ticket ID and Priority value, the graduation class (3a orthogonal-gate or 3b atomic-cohort), and the user's Rule 4 verdict (or AFK-queued status if applicable).

   **Rule 4 judgement locus (P308, 2026-06-03)**: the `wr-risk-scorer-evaluate-graduation` evaluator script implements ONLY Rule 1a join + Rule 2 VP carve-out + Rule 3b cohort grouping (`packages/risk-scorer/scripts/evaluate-graduation.sh` lines 19-22 disclaim this explicitly). It does NOT compute release-risk and does NOT apply Rule 4 evidence-floor judgement — those are LLM-judgement surfaces owned by the orchestrator + the `wr-risk-scorer:pipeline` agent per ADR-044 framework-resolution boundary. Therefore evaluator `status=resolved` means *"the ticket-join succeeded and the entry is not VP-blocked"* only — it is **necessary but not sufficient** for graduation. The orchestrator MUST evaluate Rule 4 evidence-floor (ADR-061 Rule 4 per-class shape) as a separate step before any `git mv`. Routing per `AskUserQuestion` availability: interactive surfaces per-held-entry `AskUserQuestion` (Graduate / Defer / Reject) with the inline per-entry evidence summary (P350 brief-before-ID discipline — never bare opaque P/ADR IDs); AFK queues a per-held-entry `outstanding_question` to `.afk-run-state/outstanding-questions.jsonl` (P352 / ADR-013 Rule 6 universal queue-and-continue default) and does NOT graduate. See work-problems SKILL.md Step 6.5 cohort-graduation pre-check step 2a for the full contract. The prior "auto-graduate on `status=resolved` per ADR-013 Rule 5 + ADR-061 Rule 5 silent proceed" framing is superseded by P308: ADR-013 Rule 5 silent-proceed covers *policy-authorised* outcomes; Rule 4 evidence-floor is the LLM-owned precondition that admits *evaluation* of the policy criterion. ADR-061 Rule 5 still authorises the graduation *intent* — but ADR-061 Rule 4 is the precondition that authorises the *evaluation*, and Rule 4 is not a silent surface.

   **Evidence-based, not time-based (P246 refined framing, 2026-05-17)**: the graduation criterion is **positive evidence that the surface works as desired** (defined per-class in ADR-061 Rule 4 — gate-fire log entries, detector firings, auto-fix commit log, session-trail entries). Calendar predicates (`≥7 days in-repo dogfood`, `on or after <date>`) are NEVER a primary graduation trigger and were rejected as a defective contract clause per user direction 2026-05-17: *"Dogfooding makes sense, but it shouldn't be time based, it should be until we are happy that it's working as desired."* + *"Why are we waiting? That seems to go against the principles if you ask me."* Time tracks nothing relevant to graduation: when evidence is met BEFORE a calendar threshold, the calendar artificially delays graduation; when evidence is negative (e.g. recurring hook misfires), a calendar expiry would falsely promote graduation. Per-class evidence shapes (Rule 4) are the actual graduation signal. Existing per-entry `Currently held` reinstate-trigger lines authored before the P246 framing are retained as at-hold-time historical contracts (not retroactively rewritten); new entries SHOULD phrase their reinstate criterion as evidence-of-working-as-desired only, citing the Rule 4 class-specific shape.
