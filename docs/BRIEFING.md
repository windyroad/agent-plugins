# Project Briefing

## What You Need to Know

- **Plugin hooks run from the marketplace cache**, not from source. After fixing hook code, you must: push, update marketplace (`claude plugin marketplace update windyroad`), reinstall (`claude plugin install ... --scope project`), and restart Claude Code.
- **Skill invocation names are `{plugin-name}:{skill-dir-name}`**. The `name` field in SKILL.md is display-only. Directory names must not contain colons.
- **Four edit gates fire on every edit**: architect review, JTBD review, WIP risk assessment, and TDD enforcement. Each requires its own agent delegation before the edit is allowed. Plan for this overhead. JTBD was broadened to all project files (ADR-007, superseded by ADR-008).
- **GITHUB_TOKEN pushes don't trigger pull_request events**. The release preview uses `workflow_run` trigger instead. If adding new PR-triggered workflows for changesets PRs, use the same pattern.
- **npm won't overwrite a published version**. Preview publishes use pre-release suffixes (e.g., `0.1.2-preview.13`) to avoid blocking `changeset publish` from publishing the clean version to `latest`.
- **Risk appetite is Low (4)**. Changes scoring Medium (5+) need explicit acknowledgement. See `RISK-POLICY.md`.
- **The `push:watch` script** in root `package.json` is the sanctioned way to push — it runs `git push` then watches the CI run. The git-push-gate hook blocks bare `git push`.
- **Never run `changeset version` locally.** The release pipeline handles versioning. To release: push with `npm run push:watch`, the pipeline creates a release PR, merge it with `npm run release:watch`.
- **Install at project scope** (`--scope project`) to avoid breaking other active projects. ADR-004 documents this.
- **Discord plugin setup**: Use `/discord:configure <token>` to save the bot token (stored at `~/.claude/channels/discord/.env`). Restart with `--channels plugin:discord@claude-plugins-official` to connect. Pair via DM, then lock down with `/discord:access policy allowlist`.
- **Each repo should have its own Discord bot** for wr-connect. Name it after the org/repo so sessions are distinguishable in Discord. One shared bot = all sessions look identical.
- **`.env` may be a 1Password FIFO** (named pipe). Never `cat >` to it. Use `.env.tpl` with `op://` references and `op inject -i .env.tpl -o .env` instead.
- **Plugin renames need 4 coordinated steps**: (1) rename in repo + push + release, (2) `claude plugin marketplace update windyroad` to refresh the cache, (3) edit `.claude/settings.json` directly to swap the enabled key (`claude plugin uninstall` refuses project-scope installs), (4) restart Claude Code. Then `npm deprecate "<old-pkg>@>=0.0.0" "<msg>"` — bare `*` returns 404; needs an explicit version range.
- **1Password "Developer Environments"** (UI feature) are NOT accessible via `op` CLI (no `op env get`). To read a value into a script, fall back to a vault item or a project `.env` that already has it. Voder env vars include `NPM_AUTH_TOKEN` available in `bbstats/.env`.
- **All ADRs in `docs/decisions/` are still `.proposed.md`** — none ratified. Amendments are cheap: prefer amending over superseding. When a P-problem intersects a proposed ADR, revise the ADR rather than adding a compatibility clause.
- **Risk-scorer agents are tool-restricted to `Read + Glob`** — no `AskUserQuestion`, no `EnterPlanMode`. Sub-agents invoked via Task also cannot enter plan mode on the parent's behalf. Any "scorer asks the user" design must split scorer/orchestrator concerns: scorer emits structured markers, calling skill/primary owns the interaction. See P021.
- **ADR-002 plugin dependency graph lists `@windyroad/tdd` as standalone.** Cross-plugin features (e.g. P018 JTBD traceability, P017 shared concern-splitting helper) must update the graph explicitly; don't silently add deps.
- **Parallel Claude sessions can create P-ticket numbering collisions.** Before opening new tickets outside the skill, `git fetch origin` and check the highest P-number. Inside `manage-problem` and `create-adr` the collision guard (P043, closed 2026-04-19) now computes `max(local, origin)+1` automatically. `work-problems` also fetches origin at loop start (P040, closed 2026-04-19).
- **`changesets/action@v1`'s `version:` input defaults to running `changeset version` directly**, bypassing any `npm run version` script. To fire a custom version hook (e.g. ADR-021's plugin-manifest sync), explicitly pass `version: npm run version` in `.github/workflows/release.yml`. Observed 2026-04-19 when ADR-021's first production exercise shipped a drifted Version PR because the hook wasn't wired (P052).
- **ADR-020 auto-release is skipped inside AFK loops** (per ADR-018 Step 6.5, to avoid double-release). A fix scoped to non-AFK governance paths will NOT be exercised by `/wr-itil:work-problems` iterations. When evaluating Fix Released tickets for closure from session evidence, check which code path actually fired.
- **Verify Fix Released tickets from your own in-session evidence** before deferring to the user. CI passes, preflight runs, release pipeline behaviour, next-ID assignments, and commit-gate scoring are all directly observable. Only leave `needs user verification` for genuinely user-side exercise (interactive skill invocations not triggered in session, local-plugin-cache-lag cases where the new version hasn't been re-installed yet, and behaviours only visible in the user's other projects). **At AFK loop stop**: proactively surface close-candidates whose fix paths were exercised this session — don't wait for the user to ask. Observed 2026-04-19: P028's auto-release fired end-to-end during iter 2's drain, but the loop didn't surface P028 as a close candidate until the user prompted. See P048 + P053.
- **At AFK loop stop-condition #2, surface the outstanding design questions before emitting `ALL_DONE`.** Skipped tickets whose blockers are user-answerable (naming, direction, pacing, scope) belong in a batched `AskUserQuestion` (cap 4) so the next loop can implement against the answers without re-asking. Don't treat "needs interactive input" as terminal — it's the cue to ask. Observed 2026-04-19: 4 user-answerable questions sat unasked until the user prompted; once asked, all 4 resolved in one interaction (P028 close, P049 naming, P019 direction, P051 pacing). See P053.
- **`release:watch` after `push:watch` MAY require a mid-cycle pipeline rescoring — depends on the bypass kind.** When the commit writes a `reducing-release` bypass marker (auto-applied when the commit closes a problem ticket with tests), the release gate consumes the marker without running drift detection and `release:watch` proceeds. Otherwise the `risk-gate` hook hashes pipeline state at bypass-write; post-commit+push the state hash changes and the hook blocks with `Pipeline state drift`. In that case expect the 3-step dance: `push:watch` → delegate `wr-risk-scorer:pipeline` → `release:watch`. Observed WITHOUT dance 2026-04-20 (4-changeset release via PR #36, all reducing bypasses); WITH dance 2026-04-19 iter 2 (P053) and iter 4 (P048). See P054 for the automation candidate (ADR-independent bypass-kind short-circuit).
- **ADR-022-style contract + migration split as paired commits in one AFK iteration.** When an ADR defers a mechanical migration to a follow-up commit (e.g. ADR-022 Scope lines 71-74), land both commits before pushing so the system is never pushed in an inconsistent state (new SKILL.md contract expecting `.verifying.md` glob + old `.known-error.md` files that match nothing). Iter 5 of 2026-04-19 did this correctly: commit 4e93bcf shipped the contract, immediate follow-up 72eb068 did the 13-file rename, both landed in one `push:watch`.

## What Will Surprise You

- **anthropics/claude-code#35641 is fixed** — marketplace skills show in autocomplete. The `skills` npm package workaround is no longer needed (ADR-003).
- **Each edit consumes the architect/WIP marker** — you need a fresh agent review for every blocked edit, not just one per session. The **risk-scorer, architect, and JTBD bypass markers all have a ~1800s TTL**. Long sessions WILL hit the expiry between reviews and the next edit; expect a re-review cycle after extended tool-use gaps. Observed 2026-04-19: the ADR-022 write was blocked mid-session because architect+JTBD markers from earlier P-ticket work had expired.
- **The marketplace resolves from the remote GitHub repo**, not the local working tree. You cannot install a new plugin until changes are pushed and `claude plugin marketplace update windyroad` pulls the latest. After pulling, **use `claude plugin install` (not `plugin update`)** to pick up the new code — `plugin update` compares version strings from the cache dir name (which stays at the original install version) and reports "already at latest" even when the code inside has changed. The version shown in `claude plugin list` may remain stale; verify by grepping skill content.
- **The risk-scorer PostToolUse hook uses regex dot** (`.`) not literal characters to match agent names. This was a deliberate fix — don't "correct" it back to dashes.
- **The `skills` npm package installs to 45 AI tools** (Codex, Cursor, Cline, etc.), not just Claude Code. If cross-tool distribution is ever needed again, that's the mechanism.
- **`claude plugin list` shows installs across all projects, not just the current one.** If you see N entries for a plugin, they are N different project-scope installs (correct per ADR-004), not duplicates. Check `~/.claude/plugins/installed_plugins.json` for the `projectPath` of each entry.
- **Edit gates block files outside the project** (e.g., `~/.claude/channels/discord/access.json`). Use bash to write non-project config files when gates are active.
- **Write tool fails with "File has not been read yet" after `git mv`.** Renaming a file mid-session invalidates the Read-tracking on the new path. Must `Read` the renamed file before `Write` or `Edit`.
- **Anthropic's official `skill-creator` skill ships a mature eval harness** (evals.json + dual-run with-skill/baseline + grader subagent + aggregate_benchmark + HTML viewer). Directly applicable to testing our SKILL.md documents. See P012 for how it maps to the skill-testing-harness decision.
- **The JTBD reviewer agent sometimes returns a bare verdict ("FAIL" or just a file list) without remediation reasons.** Re-query with explicit framing ("explain specifically which job conflicts and what would need to change") to get actionable guidance. See P037.
- **Skill-creator eval subagents can hit rate limits mid-batch.** Spawning 6 parallel subagents (3 with-skill + 3 baselines) risks 2-3 failing with `"You've hit your limit"`. Run fewer in parallel, or accept partial results and grade what lands.
- **Subagents spawned via the Agent tool do not inherit the parent's full skill/agent registry.** The `manage-problem` skill's commit gate hardcodes delegation to the `wr-risk-scorer:pipeline` subagent-type, which subagents typically cannot invoke — staged fixes end uncommitted. See P035/P036.
- **"Wrong Approach" (54) and "Buggy Code" (41) dominate friction types over 30 days of sessions** per the Claude Code insights report. Root cause: prompts that lump diagnose + implement + release together let autonomous loops race past diagnosis into rework. Violates JTBD-006's scope-expansion outcome and JTBD-201's evidence-before-mitigation principle. See P039.
- **External-facing text (GitHub comments, npm READMEs, issue replies, RapidAPI listings) has no voice-and-tone gate.** AI-sounding output — em-dashes, hedging, posting "keep open" on 2-year-old issues — triggers recurring "FFS" corrections. The `wr-voice-tone` plugin governs in-repo text only. See P038.
- **ADR Confirmation sections that only assert source content (e.g. "`scripts.version` contains X") can pass while the runtime integration silently fails.** ADR-021 asserted its sync script would fire via `npm run version`, but the Release workflow never passed `version:` to the action — first real release produced a drifted PR. When a fix depends on CI behaviour, add behavioural/CI-wiring assertions, not just source greps (P052 revised ADR-021's Confirmation with both).
- **`git ls-tree <ref> <path>` without `--name-only` returns lines containing blob SHAs**, which contain digit substrings that false-match `<NNN>` regex patterns. The skill-provided next-ID bash in `manage-problem` Step 3 and `create-adr` Step 3 currently has this bug — a `grep -oE '[0-9]{3}'` over raw ls-tree output sometimes picks 3-digit runs from a blob SHA. Use `git ls-tree --name-only <ref> <path>` + `sed 's|^<path>/||'` + `grep -oE '^[0-9]+'`. Observed 2026-04-20 opening P055: local_max=054 correct but origin_max=997 wrong (from a blob hash). Caught via visual sanity-check.
- **`git mv` + `Edit` + `git add` requires re-stage after the Edit.** `git mv A B` places the rename in the index. If you then use the Edit tool on B and later run `git add <other-files> && git commit`, the commit captures only the rename — the Edit to B stays unstaged and lands in the NEXT commit. Must `git add B` (or include B explicitly in the final add-list) after editing. Adjacent to the "Write fails after git mv" line above but a distinct staging-ordering trap. Observed 2026-04-19 iter 1: P054 verifying.md rename landed in 45e9c71, content edit (Status + Fix Released) only landed in iter 2's b2f1646 alongside P046's commit.
- **Validate the problem ticket's framing before asking solution-detail design questions.** When a ticket describes a specific solution (e.g. P014 described `/btw`-style stubs with backfill), the user may have a different shape in mind. First `AskUserQuestion` should confirm the framing, not dive into solution details. Observed 2026-04-20 (first incident): P014's four detail questions (stub format, backfill trigger, context escape, dispatcher) were framed around stubs; user reframed as "no stubs, auto-delegate on every invocation" in one Other-response. Observed 2026-04-20 (AFK stop-condition #2 retro): P014's three shape options (inline structured / out-of-band subagent / skill-wrapped) were framed around new patterns; user reframed again as "just use background agents" — collapsed scope from XL to L (WSJF 1.5 → 3.0) and eliminated the ADR. Other-responses keep recovering from framing misses; cost is one round of questions but the option saves the downstream skill invocation. **If the ticket has already been bounced by a framing miss once, open the next AskUserQuestion with a framing check ("Is this ticket's shape still right — pattern, skill, agent, background subagent, or something else?") before detail questions.**
- **User can escalate from AFK-stop-condition-#2 to interactive AskUserQuestion by asking.** The orchestrator's default in AFK mode is to emit an "Outstanding Design Questions" table (ADR-013 Rule 6 — the persona is away). When the user returns and the session is still alive, they can reply "ask me the questions ... use the askuserquestion tool" and the questions are batched live (cap 4 per call, multiple batches OK). Observed 2026-04-20 AFK loop iter 3 stop: 10 questions answered in 3 batches in ~5 minutes, vs potentially days of asynchronous round-trips if each answer had to prompt a new loop. The `work-problems` final summary could add an affordance line pointing the returning user at this escalation path — see codification candidate 2026-04-20.
- **Release drain fires AT appetite (≥ 4/25), not above it.** The orchestrator Step 6.5 threshold text reads "at or above appetite (4/25)". A score of exactly 4 triggers the drain. Observed 2026-04-20 iter 3: P019 commit scored 4/4/4, drain fired, @windyroad/jtbd@0.6.0 + @windyroad/itil@0.7.2 + @windyroad/architect@0.4.1 released via PR #37. If the score had been 3 no drain would have fired.
