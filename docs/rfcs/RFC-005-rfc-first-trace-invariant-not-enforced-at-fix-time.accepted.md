> **Retrofitted 2026-05-26 under RFC-006 (ADR-070); gate design corrected 2026-05-26 per P314; re-scoped RFC-first 2026-06-29 per the corrected ADR-073.** This RFC originally carried its own decisions (facets F1–F7) inline, plus a "Considered Options / Alternatives Rejected" section — exactly the P310 blind spot ADR-070 closes. The decisions it used to hold now live in the ADR ledger: gate placement → **ADR-072**, missing-RFC behaviour → **ADR-073**, RFC-first-is-unconditional → **ADR-071** (which **repudiated** the atomic-fix carve-out the original F2/F7/I13 carried). **ADR-072/073 were rewritten 2026-05-26 per P314**, then **ADR-073 was corrected RFC-first 2026-06-29** (with ADR-072 + ADR-060 I13 amended in lockstep): the 2026-05-26 "auto-create the missing RFC and never block" stance — and the 2026-06-28 P399 "author the full RFC at fix-time" amendment — were **both rejected**. The corrected decision is **RFC-first**: an RFC is a hard precondition for fix implementation; a missing RFC is **authored first** (a user-story-map decomposition, ADR-060), never fabricated as a byproduct of the fix; retrospective RFC creation is prohibited; an option-choice the existing ADR corpus does not cover escalates to a **new ADR ratified before implementation**. This RFC's mechanism prose + B-task decomposition are re-scoped to match. RFC-005 ships only the **gate mechanism** (schema + enforcement + bats); it makes no decisions of its own.

## Summary

ADR-060 I1 enforces the RFC→Problem trace at RFC capture time. The inverse direction — Problem→RFC trace at fix-time — is NOT enforced (P251). Problems routinely accrete inline `## Root Cause Analysis` → `### Investigation Tasks` → `## Fix Strategy` checklists as fix work uncovers scope, rather than the fix being scoped as an RFC.

RFC-005 ships the **mechanism** that enforces the Problem→RFC trace at fix-time: the problem-ticket reverse-trace schema, the propose-fix **RFC-first precondition gate**, the `/wr-itil:manage-problem` + `/wr-itil:work-problems` enforcement, and the behavioural coverage. The gate is **unconditional** — no atomic-fix carve-out, no effort threshold (per ADR-071). The RFC is required (must **pre-exist**) at the **propose-fix step on a Known Error** (per ADR-072, conforming to ADR-022's Known Error semantics); when a fix is proposed with no RFC, fix implementation **does not begin** — the required next action is to **author the RFC first** (a user-story-map decomposition derived from the problem, ADR-060), then implement its stories (**RFC-first**, per the corrected ADR-073). The RFC **leans on existing ADRs**; a fix approach-choice the existing corpus does **not** cover escalates to a **new ADR ratified before implementation** (the orchestrator may author the decomposition for an unambiguous / already-covered fix, but never picks an uncovered option). Retrospective RFC creation (implement-then-document) is prohibited. This holds **everywhere the gate fires** (interactive propose-fix + AFK orchestrator). "Auto-create a missing RFC and never block" — the 2026-05-26/2026-06-28 stance — is **repudiated**.

## Scope

RFC-005 extends ADR-060's I-series with the symmetric Problem→RFC trace at fix-time (the I13 invariant, in ADR-060, rewritten under P314 and re-scoped RFC-first 2026-06-29). It ships the **enforcement mechanism only** — every choice among ≥2 viable options that this RFC used to embed is now an ADR (per ADR-070):

- **Gate placement** — the RFC is required at the **propose-fix step on a Known Error**, NOT at `Open → Known Error` (a problem reaches Known Error on root cause + documented workaround alone; the fix is proposed after, and the RFC must already exist — per ADR-022). No new lifecycle state. Decision: **ADR-072**.
- **Unconditional, no carve-out** — every problem fix traces to an RFC; there is no effort threshold and no override hatch. Decision: **ADR-071** (repudiates the original F2 carve-out). Atomic fixes go through the **same** RFC as any fix — no lighter, thin, or scaled-down path; the JTBD-101 atomic-fix-adopter carve-out is removed, not relocated.
- **Missing-RFC behaviour — RFC-first** — when a fix is proposed on a Known Error with no RFC trace, fix implementation **does NOT begin**. The required next action is to **author the RFC first**: a user-story-map decomposition (ADR-060) derived from the problem's Root Cause Analysis — a deliberate pre-implementation step, **not** a skeleton fabricated during/after the fix. The orchestrator MAY author the decomposition when the fix is unambiguous **or** when its approach-choices are already covered by existing ADRs (framework-mediated derivation of already-decided direction — cite the ADRs and proceed); a fix approach-choice the existing ADR corpus does **not** cover is an independent decision (ADR-070) escalating to a **new ADR ratified before implementation** (the orchestrator does not pick it). Retrospective RFC creation is **prohibited**. This holds **everywhere the gate fires**: the interactive `/wr-itil:manage-problem` propose-fix surface AND the AFK `/wr-itil:work-problems` orchestrator (which authors-first / lawfully halts for ratification rather than auto-creating-and-proceeding). Decision: **ADR-073** (RFC-first — supersedes the auto-create-never-block stance). The cat-1 boundary (unambiguous decomposition = framework-mediated; uncovered option-choice = ratified ADR) is **ADR-044** read with **ADR-073**.
- **Load-bearing enforcement** — the propose-fix RFC-first precondition gate ships as real mechanism on day one, not advisory prose: it **refuses / routes to RFC-authoring** when no RFC pre-exists (it does not emit an auto-create directive and exit-0-proceed). Application of **ADR-051** (load-bearing-from-the-start).
- **Story-map shape** — the authored RFC is **stories in a user story map** (ADR-060), not a `Scope`+`Tasks` prose blob; I13 enforces the RFC trace + RFC-first ordering, and the decomposition is the RFC's content. Story-map *presence* is further assured by ADR-060 I7/I8.

The problem→RFC reverse-trace is the auto-maintained derived `## RFCs` body section on the problem ticket (single source of truth = the RFC's `problems:` array per ADR-060 I1; the literal `rfcs:` problem frontmatter originally proposed here is superseded — B2 reconciled 2026-06-17, dual-source rejected as split drift). Cardinality `0..N`: empty until a fix is proposed on the Known Error; `≥ 1` once a fix is proposed (the RFC is **authored first / pre-exists**, never auto-created as a byproduct).
